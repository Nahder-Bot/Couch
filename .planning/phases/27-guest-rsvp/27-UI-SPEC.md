# Phase 27 — Guest RSVP — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> This is a brownfield extension. The Couch warm-dark design language (Phase 9 / DESIGN-03) is locked.

## Surface Specifications

### 1. rsvp.html — Post-RSVP confirmation state (extended for Phase 27)

The existing `renderConfirmation()` function (rsvp.html lines 72–90) is extended with three new blocks
appended BELOW the existing `.rsvp-confirmation-sub` copy and ABOVE the `.rsvp-install-cta` button:

#### 1a. Aggregate count display

Position: directly below `.rsvp-confirmation-sub`, before push CTA.

HTML structure:
```html
<div class="rsvp-count-line">{N} going · {M} maybe</div>
```

CSS class `.rsvp-count-line` (add to rsvp.css):
- `font-family: 'Inter', sans-serif`
- `font-size: 13px` (`--t-meta`)
- `font-weight: 600`
- `color: var(--ink-warm)` (`#c9bca8`)
- `margin-bottom: 20px`
- `text-align: center`

Copy pattern (source: D-01 CONTEXT.md):
- Include "going" count only when > 0: "3 going · 1 maybe"
- Include "maybe" count only when > 0
- If both zero (only "no" responses): omit the count line entirely
- "no" responses are never surfaced to the guest (privacy posture, D-01)

Data source: returned in CF response as `guestCount` object: `{ yes: N, maybe: M }` (planner to confirm CF response shape includes these fields).

#### 1b. Push opt-in CTA (conditional on feature-detect)

Position: below aggregate count, above privacy footer.

Feature-detect gate (RESEARCH.md Q1):
```js
const pushSupported = 'Notification' in window && 'serviceWorker' in navigator && 'PushManager' in window;
```

Render the push CTA block ONLY when `pushSupported === true`. On iOS non-PWA (the majority of iPhone guests), this block is completely absent — the confirmation screen is complete without it.

HTML structure when push IS supported:
```html
<div class="rsvp-push-block">
  <div class="rsvp-push-copy">Get reminded when the party starts?</div>
  <button class="rsvp-push-btn" id="rsvp-push-btn" type="button" aria-label="Get push reminders for this watchparty" onclick="requestPushOptIn()">
    Remind me
  </button>
</div>
```

CSS class `.rsvp-push-block` (add to rsvp.css):
- `background: #25201a` (`--surface-2`)
- `border: 1px solid rgba(245,237,224,0.10)` (`--border`)
- `border-radius: 12px`
- `padding: 16px`
- `margin-bottom: 20px`
- `text-align: center`

CSS class `.rsvp-push-copy`:
- `font-family: 'Inter', sans-serif`
- `font-size: 15px`
- `color: var(--ink-warm)` (`#c9bca8`)
- `margin-bottom: 12px`

CSS class `.rsvp-push-btn`:
- Mirrors `.rsvp-install-cta` base style (rsvp.css line 49)
- `display: inline-block`
- `padding: 12px 24px`
- `background: var(--surface)` (`#1c1814`)
- `color: var(--ink)` (`#f5ede0`)
- `border: 1px solid rgba(232,160,74,0.45)` (accent-tinted border — signals primary action)
- `border-radius: 999px` (`--r-pill`)
- `font-family: 'Inter', sans-serif`
- `font-weight: 700`
- `font-size: 15px`
- `min-height: 44px` (touch target)
- `cursor: pointer`
- `:hover`: `box-shadow: 0 0 0 4px rgba(232,160,74,0.12)` (warm glow, exactly once per screen per BRAND.md §8)

Post-tap states for push button (replace button with inline copy, no modal):

State A — permission granted, subscription success:
```html
<div class="rsvp-push-done"><em>You're on the list.</em></div>
```
- Instrument Serif italic, 17px, `--ink-warm`, text-align center

State B — permission denied:
```html
<div class="rsvp-push-blocked">Notifications blocked in your browser settings.</div>
```
- Inter 13px, `--ink-dim`, text-align center

State C — push subscribe failed (network/VAPID error):
```html
<div class="rsvp-push-err">Couldn't set up reminders. Ask {hostName} to message you directly.</div>
```
- Inter 13px, `--ink-dim`, text-align center (uses `.rsvp-push-copy` color, not `.rsvp-error` red — this is not an RSVP error)

ARIA: `<button type="button">` with explicit `aria-label="Get push reminders for this watchparty"`. When button is replaced by post-tap state, the replacement `<div>` gets `role="status"` so screen readers announce it (live region implicit via role=status).

#### 1c. "Push not supported on this device" state

Do NOT render this as a fallback when `pushSupported === false`. Per RESEARCH.md Q1 risk analysis,
iOS non-PWA guests (the majority) should see a clean, complete confirmation screen — not an error or
fallback message. The absence of the push block IS the graceful degradation.

Exception: if the user taps a push button that SOMEHOW appears (defensive coding edge case), and
push then fails at the API call level (not feature-detect), show State C above inline.

#### 1d. Privacy footer (D-05)

Position: below push opt-in block (or below aggregate count if push block absent), above existing
`.rsvp-install-cta` button.

HTML structure:
```html
<p class="rsvp-privacy-footer">
  By tapping going/maybe/no, you agree to our <a href="/privacy" class="rsvp-privacy-link">Privacy Policy</a>.
</p>
```

CSS class `.rsvp-privacy-footer` (add to rsvp.css):
- `font-family: 'Inter', sans-serif`
- `font-size: 10px` (`--t-micro`)
- `color: var(--ink-dim)` (`#847868`)
- `text-align: center`
- `margin-bottom: 20px`
- `line-height: 1.5`

CSS class `.rsvp-privacy-link`:
- `color: var(--accent)` (`#e8a04a`)
- `text-decoration: underline`
- No `:hover` transform — small text, hover is desktop-only concern

Note: D-05 says footer renders on the INVITE form, not only post-RSVP. The planner must also inject
`.rsvp-privacy-footer` into `renderInviteForm()` below the `.rsvp-actions` div. Same CSS class; same copy.

#### 1e. "Your RSVP was removed by the host" eviction state (D-07)

Triggered by: 30s poll returns `{ revoked: true }` from `rsvpStatus` CF.

Render function `renderRevoked()` replaces `#rsvp-content` innerHTML:

```html
<div class="rsvp-evicted">
  <div class="rsvp-evicted-title"><em>Your RSVP was removed by the host.</em></div>
  <div class="rsvp-evicted-sub">Reach out to them directly if you think this is a mistake.</div>
</div>
```

CSS class `.rsvp-evicted` (mirrors `.rsvp-expired` in rsvp.css line 53):
- `text-align: center`
- `padding: 60px 20px`

CSS class `.rsvp-evicted-title`:
- `font-family: 'Instrument Serif', 'Fraunces', serif`
- `font-style: italic`
- `font-size: 22px` (`--t-h2`)
- `color: var(--ink-warm)` (`#c9bca8`) — warm neutral, not red (revocation is not a system error)
- `margin-bottom: 12px`

CSS class `.rsvp-evicted-sub`:
- `font-size: 13px` (`--t-meta`)
- `color: var(--ink-dim)` (`#847868`)

Accessibility — focus management on polling-triggered state change:
When `renderRevoked()` fires during a poll cycle, the page content changes without user interaction.
Move focus to the evicted-title element so screen readers announce the change:
```js
function renderRevoked() {
  contentEl.innerHTML = '...'; // as above
  const titleEl = contentEl.querySelector('.rsvp-evicted-title');
  if (titleEl) { titleEl.setAttribute('tabindex', '-1'); titleEl.focus(); }
}
```
This covers the RESEARCH.md Q1 UAT requirement (RSVP-27-09): the state change is surfaced both visually and to assistive technology within one 30s poll cycle.

Similarly for `renderClosed()` (wp.rsvpClosed = true):

```html
<div class="rsvp-closed">
  <div class="rsvp-closed-title"><em>RSVPs are closed for this party.</em></div>
  <div class="rsvp-closed-sub">The host has stopped accepting new RSVPs.</div>
</div>
```

CSS: reuses `.rsvp-expired` style (text-align center, 60px 20px padding). Title uses `.rsvp-expired-title` class (Instrument Serif italic 22px ink-warm). Sub uses `.rsvp-expired-sub` class (13px ink-dim). No new CSS needed — same visual treatment as expired.

### 2. app.html — Watchparty roster guest chips (D-04, D-07)

#### 2a. Guest chip baseline

Guest entries in `wp.guests[]` are rendered as chips appended to the existing `.wp-participants-strip`
(rendered by `renderParticipantTimerStrip()`). The existing member chip is `.wp-participant-chip` (css/app.css
line 769). Guest chips reuse the same base class with a `.guest` modifier.

HTML structure (new guest chip, mirrors `renderParticipantTimerStrip` pattern lines 12196–12203):
```html
<div class="wp-participant-chip guest" data-guest-id="{guestId}">
  <div class="wp-participant-av" style="background:#5a8a84;" aria-hidden="true">{initial}</div>
  <div class="wp-participant-info">
    <div class="wp-participant-name">
      {displayName}
      <span class="chip-badge badge-guest">guest</span>
    </div>
    <div class="wp-participant-time" data-role="pt-time">{response}</div>
  </div>
  {removeControl}
</div>
```

Display name rule (D-04, exactly as locked in CONTEXT.md):
- `if (familyMemberNames.has(guestName))` → render: `"{guestName} (guest)"` + apricot pill
- `else` → render: `"{guestName}"` + apricot pill (pill always present; suffix conditional)

`familyMemberNames` = `new Set(state.members.map(m => m.name.trim().toLowerCase()))` — lowercase comparison.
Display of the name itself preserves original casing from `wp.guests[i].name`.

Response label in `.wp-participant-time` slot:
- `yes` → "Going"
- `maybe` → "Maybe"
- `no` → "Not coming"
- absent/undefined → "Invited" (no RSVP yet)

Avatar color for guests: fixed teal `#5a8a84` (derived from the `badge-guest` teal `#7cb4a9`, darkened for
avatar use — same hue family, distinguishable from member colors). Do NOT randomize per-guest — guests have
no persistent color assignment. All guests use the same avatar color; the `badge-guest` pill is the primary
identity signal.

CSS additions to app.css (append after existing `.wp-participant-chip` block, css/app.css line 771):
```css
/* Phase 27 — Guest chip variant */
.wp-participant-chip.guest {
  border-color: rgba(124, 180, 169, 0.35); /* badge-guest teal, 35% opacity */
}
.wp-participant-chip.guest .wp-participant-av {
  background: #5a8a84; /* fixed teal avatar for all guests */
}
```

The `.chip-badge.badge-guest` style is already defined at css/app.css line 2335:
```css
.badge-guest { background: #7cb4a9; color: #0f1f1c; }
```
No changes needed to this rule. Phase 27 uses it as-is.

#### 2b. Remove control — kebab menu (Claude's Discretion, per CONTEXT.md)

Decision: use a kebab (three-dot) button on the guest chip, not long-press. Rationale:
- The existing `removeMember()` in app.html uses an inline "Remove" button (js/app.js line 6523), not long-press.
- Long-press on a chip in the participants strip would conflict with the existing "I started on time" control.
- Kebab is consistent with app.html's pattern of per-row action disclosure.
- Kebab is host-only: visible only when `state.me && state.me.id === wp.hostId`.

HTML for `{removeControl}` slot (host-only, injected into guest chip):
```html
<button class="wp-guest-kebab" type="button"
  aria-label="Guest options for {displayName}"
  onclick="openGuestMenu('{guestId}', event)">
  &#8942;
</button>
```

CSS class `.wp-guest-kebab` (add to app.css):
- `background: transparent`
- `border: none`
- `color: var(--ink-dim)` (`#847868`)
- `font-size: 18px`
- `cursor: pointer`
- `padding: 8px` (creates 44×44 touch target with the chip padding: 6px top/bottom on chip + 8px button padding = 22px height, so set `min-width: 44px; min-height: 44px; display: grid; place-items: center`)
- `:hover`: `color: var(--ink)` (`#f5ede0`)

Guest options menu (shown on kebab tap, per `openGuestMenu()`):
- Single item: "Remove guest" — text `color: var(--bad)` (`#c44536`) to signal destructive
- No confirmation modal required for kick — the revoke is soft-delete (D-07), not destructive data loss
- Implemented as a small popover anchored to the kebab button, consistent with existing app.html dropdown patterns

Destructive action copy for guest removal:
- Menu item label: "Remove {name}"
- Toast on success: "Removed." (brief, `.flashToast` warm style, existing utility)
- Toast on failure: "Couldn't remove {name}. Try again."

#### 2c. Aggregate count chip on wp banner / lobby

The existing `.wp-banner-meta` (css/app.css line 733) shows party time and status. Phase 27 adds
guest count to the wp banner when `wp.guestCount > 0`.

Copy pattern (source: CONTEXT.md Specifics, D-01):
- "5 going · 2 guests"
- Only rendered when `wp.guestCount >= 1`
- Uses denormalized `wp.guestCount` field (D-01)
- Appended to existing `.wp-banner-meta` text, separated by ` · `

No new CSS class needed — appended text inherits `.wp-banner-meta` style:
`font-size: var(--t-meta)` (13px), `color: var(--ink-dim)` (`#847868`).

## Accessibility Contract

### Push CTA

- Element type: `<button type="button">` (not `<a>`) — triggers an async JS action, not navigation
- `aria-label`: "Get push reminders for this watchparty"
- Post-tap state (success/blocked/fail): replaced `<div>` with `role="status"` — screen reader announces the replacement without focus trap or modal

### Push not supported (iOS non-PWA)

No announcement needed — the CTA block is simply absent. Nothing to announce. Static rendered confirmation page is complete. No live region, no hidden text. (RESEARCH.md Q1 recommendation: treat absence as the primary path.)

### Eviction state focus management

When `renderRevoked()` or `renderClosed()` fires during a 30s poll cycle (background timer, no user gesture):
- Set `tabindex="-1"` on the title element and call `.focus()` immediately after render
- This moves focus to the new content and triggers screen reader announcement
- The tabindex="-1" is non-interactive — it only enables programmatic focus

### Guest chip kebab button

- `aria-label="Guest options for {displayName}"` (dynamic per guest)
- `type="button"` to prevent form submission
- 44×44px minimum touch target (explicit `min-width`/`min-height` on `.wp-guest-kebab`)

### Aggregate count display

- The `.rsvp-count-line` div is static text, no interaction. No ARIA role needed.
- Screen readers read it in natural document order (below confirmation title, before push CTA).
