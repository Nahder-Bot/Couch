### Task 2 — 3-way firestore.rules lockstep (commits queuenight `c4aaedb` + couch `ab1279d`)

- New `match /couchPings/{pingId}` block inserted BETWEEN `match /watchparties/{wpId}` and `match /intents/{intentId}` blocks (preserves existing rule-file ordering). Block contents:
  ```
  allow read: if isMemberOfFamily(familyCode);
  allow create: if attributedWrite(familyCode)
    && request.resource.data.senderId is string
    && request.resource.data.senderId.matches('^m_[A-Za-z0-9_-]+$')
    && request.resource.data.recipientId is string
    && request.resource.data.recipientId.matches('^m_[A-Za-z0-9_-]+$')
    && request.resource.data.senderId == request.resource.data.memberId;
  allow update: if false;
  allow delete: if false;
  ```
- The `senderId == memberId` clause is the T-15.4-01 spoofing mitigation: validAttribution already pins `memberId == request.auth.uid`, so transitively `senderId == request.auth.uid` — Member A cannot write a ping claiming to be from Member B (which would surface a fake "B wants you" push to the recipient).
- The `recipientId.matches('^m_[A-Za-z0-9_-]+$')` clause is the T-15.4-02 regex-injection defang — same SEC-15-1-02 anchor pattern that protects memberId/managedMemberId in validAttribution, now extended to recipientId (which is a NEW field not covered by the helper).
- `allow update/delete: if false` — single-shot ephemeral doc; the CF (admin SDK bypasses rules) owns lifecycle.
- All 3 firestore.rules files **byte-identical post-edit** at 36445 bytes — verified via `node` byte-compare (`a !== b || b !== c → process.exit(1)`).
- `couch/tests/firestore.rules` is **gitignored** by repo convention (Phase 15.1 setup) — `tests/package.json` pretest hook runs `require('fs').copyFileSync('../firestore.rules','firestore.rules')` before each `npm test` to keep it in sync.
