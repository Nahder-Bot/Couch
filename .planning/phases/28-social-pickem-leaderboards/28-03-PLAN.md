<objective>
Wave 2 backend — settlement + reminder Cloud Functions + 3-map notification lockstep extension.

Purpose: Bring the asynchronous settlement + reminder loop online. Picks submitted via the
Plan 28-05 client UI are settled by `gameResultsTick` once their underlying game finishes
(TheSportsDB final score for team-sport / soccer leagues; Jolpica race results for F1).
Leaderboards are updated atomically per settle. Members who haven't picked at T-15min get
a reminder push via `pickReminderTick`. All three new push event keys (`pickReminder`,
`pickResults`, `pickemSeasonReset`) are added to the 3-map notification stack in a SINGLE
task to prevent the partial-deploy drift window flagged by Pitfall 5.

**Critical scope deltas vs the original (now-deleted) Plan 28-03 + REVIEWS amendments:**

1. **UFC settlement DROPPED.** Per CONTEXT.md D-17 update 2 (2026-05-04), UFC was removed
   from Phase 28 v1. `gameResultsTick` has NO UFC branch.

2. **F1 source = Jolpica, NOT TheSportsDB.** F1 race results come from
   `https://api.jolpi.ca/ergast/f1/{year}/{round}/results/` (free, no key, verified 2026-05-04).

3. **gameResultsTick has 2 settlement branches** (down from the original 4):
   - **TheSportsDB branch** for `team_winner` + `team_winner_or_draw`.
   - **Jolpica branch** for `f1_podium`.

4. **REVIEWS Amendment 1 / HIGH-2 — Counter ownership locked.** `gameResultsTick` is the
   SOLE writer of `picksTotal`, `picksSettled`, `picksAutoZeroed`, `lastPickAt`.
   `picksTotal` increments **exactly once per pick** on first processing (settle path OR
   auto-zero path; the unknown-pickType skip path does NOT increment because the pick was
   never the CF's responsibility). `lastPickAt` is set from `pick.submittedAt`. Client
   NEVER writes leaderboard (Plan 04 rule denies).

5. **REVIEWS Amendment 2 / HIGH-4 — Backend UFC defense.** Unknown pickType (anything
   not in `{team_winner, team_winner_or_draw, f1_podium}`) is `continue`d BEFORE the
   transaction. Pick state is NOT mutated, leaderboard is NOT touched. The original plan
   marked unknown picks as `state: 'settled'` with 0 pts — which would have inflated
   `picksSettled` denominators. New behavior: phantom UFC pick sits in Firestore at
   `state: 'pending'` forever (Plan 04 rule prevents new UFC picks from being created
   anyway via the `pickType` allowlist constraint).

6. **REVIEWS Amendment 3 / MEDIUM-5 — Branch-aware AUTO_ZERO_GRACE_MS.** Replaces the
   global 1-hour grace with a `GRACE_BY_PICKTYPE` map:
   - `team_winner`: 3h (NBA/NFL/MLB/NHL/etc. games run ~2-3h max — generous tolerance for
     TheSportsDB final-score lag).
   - `team_winner_or_draw`: 3h (soccer 90-120min + extra time + TSD lag).
   - `f1_podium`: 24h (Jolpica/Ergast can lag the podium by hours on race day; per Codex's
     stronger-recommendation read, an F1 pick should NEVER be auto-zeroed just because the
     source is slow — 24h is effectively "Jolpica had a really bad day" tolerance).

7. **REVIEWS Amendment 4 / MEDIUM-10 — Per-tick fetch cache.** A `Map<string, finalGame>`
   keyed by `${leagueKey}:${gameId}` (team-sport) and `${f1Year}:${f1Round}` (F1) caches
   each score fetch for the duration of one tick. 5 family members all picking the same
   NBA game = 1 `fetchTsdScore` call per tick instead of 5.

8. **REVIEWS Amendment 5 / HIGH-3 — Composite indexes declared.** New
   `firestore.indexes.json` with two collectionGroup indexes:
   - `watchparties` × `(mode, hostFamilyCode)` for pickReminderTick
   - `picks` × `(state, gameStartTime)` for gameResultsTick
   Plan 06 deploys via `firebase deploy --only firestore:rules,firestore:indexes`.

9. **Failure-mode is per-branch fail-soft.** If Jolpica is down or returns 5xx for an F1
   race, that pick stays `pending` for the next tick. Team-sport / soccer picks continue
   to settle normally.

export function scorePick(pick, finalGame) {
  if (!pick || !finalGame) return 0;
  if (finalGame.state !== 'post') return 0;
  try {
    const sel = pick.selection || {};
    switch (pick.pickType) {
      case 'team_winner':
        return sel.winningTeam === finalGame.winningTeam ? 1 : 0;
      case 'team_winner_or_draw':
        return sel.result === finalGame.result ? 1 : 0;
      case 'f1_podium':
        return (sel.p1 === finalGame.p1 && sel.p2 === finalGame.p2 && sel.p3 === finalGame.p3) ? 1 : 0;
      // NOTE: ufc_winner_method intentionally OMITTED here per D-17 update 2 (UFC dropped from v1).
      default:
        return 0;
    }
  } catch (e) { return 0; }
}
