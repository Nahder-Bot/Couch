<objective>
Wire Phase 24's user-visible surface end-to-end across `js/app.js`, `app.html`, `css/app.css`. Plan is split into FIVE sub-tasks (was four; one added for REVIEWS H1 DOM scaffold split).

**The H1 fix is the most important change.** Original Plan 03 placed the player INSIDE `#wp-live-content`, which `renderWatchpartyLive` repaints via `el.innerHTML = ...` on every reaction post / timer tick / wp record update. The player would be destroyed and recreated on every state change — catastrophic. The fix splits the modal into two siblings:

- `#wp-video-surface` — **persistent**, only touched by `attachVideoPlayer` / `teardownVideoPlayer`. Never re-rendered.
- `#wp-live-coordination` — **re-renderable**, absorbs everything `renderWatchpartyLive` currently writes.

This refactor ripples into Tasks 3.1 (DOM scaffold + UI-SPEC update), 3.3 (renderWatchpartyLive targets coordination only), 3.4 (lifecycle targets video-surface).

**What ships in this plan (post-revision):**
1. **DOM scaffold split (REVIEWS H1):** Update `app.html:1176-1177` to split `#wp-live-content` into `#wp-video-surface` + `#wp-live-coordination` siblings inside `.wp-live-modal`. Update UI-SPEC.md inline to document the split.
2. **Three distinct URL fields (REVIEWS M3):** `#wp-video-url-movie` (in `#wp-start-modal-bg`), `#wp-video-url-game` (in `#game-picker-modal-bg`), `#wp-video-url-sport` (in `#sports-picker-bg`). Each flow reads its own ID.
3. **CSS rules (UI-SPEC §1 + §2):** `.wp-video-frame`, `.wp-video-surface` (sized 0 when empty), `.field-help`, `.field-error`, `.field-invalid`, `.wp-live-header--has-player`, `.wp-video-error`.
4. **Persistence (D-04 + VID-24-06/07/08):** confirmStartWatchparty + confirmGamePicker + scheduleSportsWatchparty all parse + persist videoUrl + videoSource. **REVIEWS C1: mixed-content HTTP MP4 warning toast.**
5. **Render branches (D-01 + D-02 + VID-24-01/03/11):** renderWatchpartyLive computes coordination HTML (NOT player HTML — player is owned by attach/teardown). Player HTML is built by `attachVideoPlayer` writing into `#wp-video-surface`.
6. **DRM-hide branch (D-03 + VID-24-05):** `#wp-video-surface` stays empty when DRM-only.
7. **Lifecycle (RESEARCH Pitfall 6):** attachVideoPlayer renders into `#wp-video-surface`, runs YT div→player upgrade per REVIEWS H3, attaches NAMED event handlers (REVIEWS H5), seeks late-joining non-hosts via seekToBroadcastedTime (REVIEWS M1). teardownVideoPlayer removes handlers by ref, clears surface.
8. **Broadcast (VID-24-09 + VID-24-10):** host-only `makeIntervalBroadcaster(VIDEO_BROADCAST_INTERVAL_MS, ...)` writes extended schema (REVIEWS C2: currentTimeSource, durationMs, isLiveStream). Live-stream gate is per-sample (REVIEWS M4).
9. **Error overlay (VID-24-14):** renderPlayerErrorOverlay + delegated retry click (REVIEWS H4 — link is wired, not decorative).

This plan implements VID-24-01, 03, 06, 07, 08, 09, 11, 14, 15, 16, 17. Plan 04 ships sw.js bump + Firestore rules verification (REVIEWS M2) + deploy + UAT.

After this plan completes, the watchparty live modal is fully functional end-to-end pre-deploy AND survives reactions/timer ticks without resetting the player (the H1 fix).
