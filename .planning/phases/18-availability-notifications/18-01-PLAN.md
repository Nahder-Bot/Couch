**Why a daily scheduled CF (D-01, RESEARCH risk 2):** TMDB provider data lags
JustWatch/studio reports by 24-72h; daily cadence absorbs short-lag false positives
without spamming. Hourly = noisy. Weekly = misses the moment-of-availability win.

**Why `t.queues` non-empty AND `t.watched !== true` (D-06 + Specifics #3):** Titles
with zero queue interest aren't in anyone's pack of intent. Already-watched titles
don't deserve "now available on Max" pings — a household member already saw it.

**Why subscription bucket only (D-07):** rent/buy is always available — only "free
with my subscription" is the moment that matters for the user's pack.

**Why setBy: 'system' marker (D-08):** Distinguishes this CF write from
`writeAttribution()` user writes per Phase 15.1 attribution pattern; the field is
informational (admin SDK bypasses rules anyway), but downstream observability +
future audit hooks benefit from the marker.

**Why two batch shapes (D-14 + D-15):** Single title gets a clean direct body
("Dune just hit Max for your household."). Multi-title within a tick gets a roll-up
("3 titles your couch wants…") capped at 3 surfaced titles to fit a push body
without truncation.

**Why TMDB 429 → abort tick rather than exponential backoff (D-04):** Daily cadence
is forgiving — losing a tick costs 24h, not data. Adding a backoff loop would risk
multi-tick lock contention for negligible benefit at current scale (RESEARCH risk 4).

- "just hit" — vivid, directional, no banned terms

- "your couch wants" — BRAND-voice (warm, agentive, family-tapping-shoulder)
- "are now watchable" — neutral, no theatrics
- No banned words ("buffer", "queue", "sync", "lag", "latency", "playback")

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Cloud Scheduler → providerRefreshTick CF | Trusted (GCP-managed scheduler). No untrusted input. |
| TMDB API response → providerRefreshTick body | Untrusted external data. Provider names are normalized via the static map (defangs unexpected provider names) and capped at 12 entries (slice(0, 12)) so a malicious TMDB response cannot inject unbounded data into the family doc. |
| providerRefreshTick → Firestore title doc writeback | Admin SDK bypasses rules; setBy:'system' marker distinguishes from user writes for downstream observability. |
| providerRefreshTick → sendToMembers → FCM | Push body computed server-side from t.name + canonical brand names; member.services is a member-controlled string array but only the INTERSECTION with TMDB-derived brands is used in the body, so a member writing a malicious service string ('javascript:alert' etc.) cannot reach the push body unless TMDB also returns it as a provider name (it won't). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-18-01 | Tampering (TMDB returns bogus provider data) | fetchTmdbProvidersServerSide | mitigate | Provider names are normalized through the static map (only canonical brands pass); the slice(0, 12) cap bounds list size; the diff is intersection-based so only brands ALREADY in the canonical set can match a member's services. A bogus brand from TMDB can land in t.providers but cannot reach a push body unless a member happens to have it in their services list, which they wouldn't. |
| T-18-02 | DoS (TMDB rate limit triggers cascade failure) | per-tick TMDB calls | mitigate | 429 detection short-circuits the tick (sets aborted=true), per-doc try/catch isolates per-title failures. Daily cadence + 50-title cap = ~12.5s of TMDB traffic / day at full throttle, well under the rate limit. |
| T-18-03 | DoS (large family with many titles starves later families) | per-family loop ordering | accept | Per-tick cap is global (50 titles total across all families). Families processed in Firestore-list order. At current scale (a handful of families) this is fine. If user count grows 10x, revisit with per-family round-robin cursor. |
| T-18-04 | Information disclosure (push body leaks t.name to wrong recipients) | buildPushBody / sendToMembers | mitigate | Recipients filtered by member.services intersection — only members whose pack matches the new brand are notified. titleName is a public TMDB title, not user-generated content; no PII risk. Member.services is a per-member preference set by that member, not cross-member-leakable. |
| T-18-05 | Spoofing (system writes attributed as user writes) | title doc setBy field | mitigate | setBy:'system' marker distinguishes CF writes from writeAttribution()-stamped user writes. Future audit/observability hooks can filter on this. Rules don't enforce setBy (admin SDK bypasses rules anyway), but the marker preserves traceability. |
| T-18-06 | Configuration (TMDB_KEY missing in .env) | env-var read at module load | mitigate | TMDB_KEY env var read once at module load; if empty, providerRefreshTick logs a warning and short-circuits without attempting fetches — no crash, no partial data corruption. |
