    }

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client → Firestore (users/{uid}) | User-controlled UI writes pref booleans + quiet-hours strings to their own user doc |
| Firestore → Cloud Function (sendToMembers) | Function reads recipient's `users/{uid}.notificationPrefs` to decide whether to push (Phase 6 enforcement, unchanged) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-12-01-01 | Tampering | client → Firestore users/{uid}.notificationPrefs writes | mitigate | Existing Phase 5 firestore.rules already restrict `users/{uid}` writes to `request.auth.uid == uid`. This plan adds no new write paths — only refactors UI markup that calls existing writers `updateNotificationPref` and `updateQuietHours`. No new attack surface. |
| T-12-01-02 | Information disclosure | quiet hours string format | accept | Stored values are HH:mm + IANA tz only — no PII beyond what the user already provides via the time picker. Firestore rule scope limits read to self. |
| T-12-01-03 | Spoofing | UI key vs server key alias | mitigate | `NOTIF_UI_TO_SERVER_KEY` is a const Object.freeze with a 6-key allowlist. `savePerEventToggle` short-circuits if `uiKey` is not in the map (unknown-key log + early return). The server-side `sendToMembers` `eventType` parameter is set by Cloud Function call sites, not by client — so a malicious client cannot inject an arbitrary key into push delivery (it can only flip prefs for the 8 known server keys via existing writers). |
| T-12-01-04 | DoS via toggle storm | rapid checkbox toggling | accept | Each toggle write is a single-field setDoc with merge — Firestore handles ≥1 write/sec per doc easily. flashToast is throttled by user gesture rate (no DoS risk). Phase 6 server enforcement is read-once-per-push. |
| T-12-01-05 | Repudiation | toggle change without audit log | accept | Per CLAUDE.md the app explicitly chose not to log preference changes (small family scale, no audit requirement). `updatedAt` Firestore implicit ts is sufficient if ever needed. |

No high-severity threats. No new attack surface introduced.
