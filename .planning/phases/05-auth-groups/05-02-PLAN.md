    **Step A — Commit the domain-association file:**
    1. User has downloaded `apple-developer-domain-association.txt` from Apple during Task 1.
    2. Create the directory if missing: `mkdir -p ../queuenight/public/.well-known/`.
    3. Place the file at `../queuenight/public/.well-known/apple-developer-domain-association.txt`.
    4. File MUST NOT have any wrapper/framing — it is raw Apple-provided text.

    **Step B — Inspect and adjust `../queuenight/firebase.json`:**
    Look for the `hosting.rewrites` array. The common failure mode (RESEARCH P9, A3) is a catch-all rewrite like:
    ```json
    { "source": "**", "destination": "/index.html" }
    ```
    that eats `/__/auth/*` and `/.well-known/*`.

    Firebase Hosting's matcher evaluates rewrites in order; earlier rewrites win. To make the auth handler + domain-association file pass through unmodified, ensure the config includes:
    ```json
    "hosting": {
      "public": "public",
      "rewrites": [
        { "source": "/__/auth/**", "destination": "/__/auth/handler" },
        { "source": "/.well-known/**", "function": null }
      ]
    }
    ```

    Actually the simpler correct config is: Firebase Hosting auto-handles `/__/auth/*` for the project's authDomain (no rewrite needed), and static files under `public/.well-known/` are served directly if no rewrite catches them first. **Required change:** if a catch-all SPA rewrite exists, exclude the two paths:
    ```json
    "rewrites": [
      { "source": "**/!(__/auth/**|.well-known/**)", "destination": "/index.html" }
    ]
    ```
    — OR (simpler) ensure pass-through entries precede any catch-all:
    ```json
    "rewrites": [
      { "source": "/__/auth/**", "destination": "/__/auth/handler" },
      { "source": "/.well-known/**", "destination": "/.well-known/:splat" }
    ]
    ```
    — whichever idiom the existing config already uses. **If firebase.json has no catch-all rewrite at all, no change is needed** — just commit the domain-association file.

    **Step C — Deploy and verify:**
    ```bash
    cd ../queuenight
    firebase deploy --only hosting
    # then verify:
    curl -s -o /tmp/apple.txt -w "%{http_code}\n" https://couchtonight.app/.well-known/apple-developer-domain-association.txt

    # expect /tmp/apple.txt content to match exactly what Apple supplied
    curl -s -o /dev/null -w "%{http_code}\n" https://couchtonight.app/__/auth/handler
    # expect: 200 (HTML response) — NOT 404
    ```

    Return to Apple Developer → Services ID → Configure → Domains and Subdomains → click "Verify" next to `couchtonight.app`. Apple fetches the domain-association file and marks verified (green checkmark). This must succeed before Firebase Auth Apple provider will work.
