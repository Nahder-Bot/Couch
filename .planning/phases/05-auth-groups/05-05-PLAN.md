<objective>
Introduce the single `writeAttribution()` helper that every Firestore write site will use to stamp `actingUid + managedMemberId + legacy memberId` onto payloads, then migrate every existing `{ memberId: state.me.id, memberName: state.me.name }` write site in `js/app.js` to use it. Update self-echo guards (`=== state.me.id` against incoming snapshot data) to dual-check against `actingUid`. This plan is the schema-attribution backbone (D-20, D-21) and keeps AUTH-05 intact: every shipped feature writes the grace-compatible shape and every real-time subscriber still filters itself out correctly.

Purpose: Without a single helper, 25+ write sites drift apart. Without dual-write, post-migration writes lose grace-window compatibility. Without updated self-echo guards, VETO-03 toast suppression breaks as soon as a user acts-as a sub-profile.
Output: `js/utils.js` gains one exported function; `js/app.js` has ~25 inline replacements and ~8 guard updates; no regressions to Tonight/Mood/Veto/Watchparty flows on the deployed site.
</objective>

         const actingUid = (state.auth && state.auth.uid) || null;
         const actingAsSubProfile = state.actingAs || null;
         const actingAsName = state.actingAsName || null;
         const payload = {
           actingUid,

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client `writeAttribution()` output ↔ Firestore rules | Rules validate `actingUid == request.auth.uid` — client cannot forge a different uid |
| `state.actingAs` mutable global ↔ write site | Per-action clear prevents cross-request contamination |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-05-01 | Spoofing | Client forges `actingUid` in payload | mitigate | Firestore rules (Plan 04) enforce `validAttribution()` — `request.resource.data.actingUid == request.auth.uid` |
| T-05-05-02 | Tampering | Client acts-as a sub-profile they don't manage | mitigate | Rules check `get(.../members/{managedMemberId}).data.managedBy == uid()` — mismatched parents denied |
| T-05-05-03 | Race | Stale `state.actingAs` from previous tap leaks into unrelated write | mitigate | writeAttribution clears actingAs AFTER snapshotting into local and BEFORE returning payload — matches VETO-03 pattern |
| T-05-05-04 | Information-disclosure | Cross-member impersonation via missed self-echo | mitigate | Dual-check updates prevent acting-as events from being suppressed as self on other devices |
| T-05-05-05 | Availability | A writeAttribution bug breaks all Firestore writes | accept | Deployed smoke test is the gate — if Tonight votes fail post-deploy, revert and debug before moving on |
</threat_model>
