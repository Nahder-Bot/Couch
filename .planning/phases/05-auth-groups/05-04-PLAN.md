<objective>
Ship the Firestore security rules that enforce the acting-uid attribution model + grace-window semantics, plus a Node rules-unit-testing script that proves the rules behave correctly for all critical scenarios (including the three narrow client-create branches Plan 06 + Plan 07 depend on). Bootstrap the `/settings/auth` Firestore doc with the grace cutoff timestamp so rules can read it live.

Purpose: Rules are the ONLY security gate for direct Firestore writes (Cloud Functions use admin SDK which bypasses rules). Without these, Plan 05's migrated write sites could be bypassed by anyone with a Firebase API key (which is public-by-design). Additionally, Plans 06 and 07 perform client-side setDoc/batch writes (first-group-create, first-member-is-owner, sub-profile create) that rules MUST permit under narrow conditions — overly-restrictive rules here would break those plans.
Output: Production rules file deployed, settings doc seeded, 10+ passing unit tests.
</objective>

Those two rules MUST be written verbatim as specified here.

    1. `rules_version = '2';` — required for Firestore rules v2 features.
    2. Helpers: `signedIn()`, `uid()`, `isMemberOfFamily(code)`, `isOwner(code)`, `isParent(code)`, `graceActive()`, `validAttribution(code)` — exactly as in RESEARCH §5.
    3. Matches:

       **`/families/{familyCode}` — family doc (DEVIATION FROM RESEARCH TEMPLATE):**
       ```
       match /families/{familyCode} {
         allow read: if isMemberOfFamily(familyCode);
         // Client may CREATE a brand-new family doc when stamping themselves as owner.
         // This is the Plan 06 submitFamily new-group path.
         allow create: if signedIn()
                       && !exists(/databases/$(database)/documents/families/$(familyCode))
                       && request.resource.data.ownerUid == request.auth.uid;
         // Updates (passwordHash, ownership transfer, rotation) go through Cloud Functions only.
         allow update: if false;
         allow delete: if false;
       }
       ```

       **`/families/{familyCode}/members/{memberId}` — member doc (DEVIATION FROM RESEARCH TEMPLATE):**
       ```
       match /families/{familyCode}/members/{memberId} {
         allow read: if isMemberOfFamily(familyCode);

         // Three narrow client-create branches:
         //   Branch A: first-member bootstrap — family doc's ownerUid is this uid
         //             (getAfter works because the family create in the same batch is already staged).
         //             Plan 06 joinAsNew first-member path triggers this.
         //   Branch B: existing member creates sub-profile they will manage.
         //             Plan 07 createSubProfile triggers this. No uid on the sub, managedBy == caller.
         //   Branch C: existing family-authed device joining as a new authed member.
         //             Plan 06 joinAsNew for non-password groups when user already has membership
         //             on a different device (unlikely but keeps symmetry).
         //             NOTE: password-protected groups MUST go through joinGroup CF — Branch C requires
         //             isMemberOfFamily(familyCode) which fails for a non-member; only the CF admin-SDK
         //             write path bypasses this.
         allow create: if signedIn() && (
           // Branch A — first-member bootstrap
           (
             getAfter(/databases/$(database)/documents/families/$(familyCode)).data.ownerUid == request.auth.uid
             && request.resource.data.uid == request.auth.uid
             && !('managedBy' in request.resource.data)
           )
           // Branch B — sub-profile creation by existing family member
           || (
             isMemberOfFamily(familyCode)
             && !('uid' in request.resource.data)
             && request.resource.data.managedBy == request.auth.uid
           )
           // Branch C — existing family-authed device adds a new authed-self member doc
           || (
             isMemberOfFamily(familyCode)
             && request.resource.data.uid == request.auth.uid
             && !('managedBy' in request.resource.data)
           )
         );

         // Updates: self, owner, or parent-of-managed-sub.
         allow update: if signedIn() && (
           resource.data.uid == request.auth.uid
           || isOwner(familyCode)
           || resource.data.managedBy == request.auth.uid
         );

         // Deletes: owner only (Plan 06 leaveFamily + owner removing member).
         allow delete: if isOwner(familyCode)
                       || (signedIn() && resource.data.uid == request.auth.uid);
       }
       ```

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client Firestore SDK ↔ Firestore backend | Rules evaluate every read/write; admin SDK (in CF) bypasses rules |
| `/settings/auth` ↔ rule helper `graceActive()` | Rules `get(...)` the doc each evaluation (cached per-request) |
| Client-side family/member CREATE ↔ rules | Three narrow create branches for members, one for family doc (new+self-owner); CF-only for updates |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-04-01 | Elevation-of-privilege | Unauthed client writes to any family doc | mitigate | Every match block's write requires `signedIn()` via helper chains; unit test #6 verifies |
| T-05-04-02 | Elevation-of-privilege | Authed but non-member writes to someone else's family | mitigate | `isMemberOfFamily(code)` helper queries `users/{uid}/groups/{code}` existence; unit tests #3 + #14 verify |

| T-05-04-05 | Tampering | Post-grace client forges legacy `memberId` to bypass uid checks | mitigate | Rule flips `graceActive()` branch off once `settings/auth.graceUntil < request.time`; test #5 verifies |
| T-05-04-06 | Info-disclosure | Client reads `families/{code}/invites/{token}` to enumerate | mitigate | `invites` match block `allow read, write: if false`; test #7 verifies |
