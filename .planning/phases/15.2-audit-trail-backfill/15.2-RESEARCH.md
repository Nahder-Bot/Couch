**What goes wrong:** A naive 05-VERIFICATION.md might try to enumerate all 109 `writeAttribution` call sites in `js/app.js` to "prove" AUTH-02 coverage. This creates a 100+ line wall of file:line references that adds zero signal and rots fast (every Phase 14/15/16 commit moves the line numbers).

**How to avoid:** In the Required Artifacts table, cite the **function definition + the count** ("`writeAttribution` function at `js/app.js:~XXX` + 109 call sites verified via `grep -c writeAttribution js/app.js`"). Include the grep command itself so readers can re-verify. Use the same pattern for the 47 token uses, the 1840 `var(--ŌĆ”)` uses, and any other "spread across the codebase" pattern.

### Pitfall 2: Phase 7 cross-repo push fan-out ŌĆö coordinate with what's actually in queuenight/functions/index.js

**What goes wrong:** 07-01-SUMMARY.md and 07-03-SUMMARY.md need to reference the watchparty Cloud Functions (`onWatchpartyCreate`, `onWatchpartyTick`, etc.) that live in `C:/Users/nahde/queuenight/functions/index.js` ŌĆö a sibling repo that is NOT git-tracked from couch (per CLAUDE.md architecture + 07-05-SUMMARY's deviation block). The SUMMARY can't cite a couch-repo commit hash for the CF; the deploy itself is the shipping surface.

**How to avoid:** When backfilling 07-01 and 07-03 SUMMARYs, follow the precedent in 07-05-SUMMARY (it already documents this exact pattern at "Issues Encountered" ŌåÆ "queuenight repo is not git-tracked"). In the new SUMMARYs, write: "Task N CF deployment shipped to queuenight-84044 us-central1; couch-repo has no commit reference for this side; deploy log evidence is the canonical shipping surface." This pattern is repeated across Phases 6, 7, 8 (all push-related work) and Phase 14 (intent CFs).

### Pitfall 3: AUTH-05 + COMP-13-01 + OPS-13-04/05/07 carry HUMAN-VERIFY tags

**What goes wrong:** A naive traceability refresh might flip these from "Pending" to "Complete" without preserving the deferred-UAT context. That loses information valuable to anyone running `/gsd-progress` who needs to know there's still a manual gate before this milestone fully closes.

**How to avoid:** Use the Phase 14 / DECI-14-* row pattern from REQUIREMENTS.md lines 307-319 ŌĆö `Complete ŌĆö UAT-DEFERRED (...)` or `Complete ŌĆö DEPLOY-DEFERRED (...)` or `Complete ŌĆö HUMAN-VERIFY pending (...)`. The descriptive suffix is part of the canonical row format for the project.

### Pitfall 4: DESIGN-01 must remain Unsatisfied in 09-VERIFICATION.md

**What goes wrong:** A planner overzealous about closing gaps might mark DESIGN-01 as "satisfied via PNG icon set" since the PNGs are wired and functional. **This is wrong.** The audit YAML at lines 39-50 is explicit: "Source-of-truth SVG absent (rebuild from source impossible)." Phase 15.3 is the one that closes DESIGN-01 by producing the SVG.

**How to avoid:** 09-VERIFICATION.md must record DESIGN-01 status as `unsatisfied` with explicit rationale ("PNG icon set ships and meets functional acceptance ŌĆö manifest + head assets wired ŌĆö but the rebuild-from-source SVG artifact is absent. Requirement closed by Phase 15.3."). Don't soften the language.

### Pitfall 5: SUMMARY frontmatter convention drift

**What goes wrong:** The audit at line 419 explicitly flags "SUMMARY frontmatter convention drift: `requirements_completed` vs `requirements_addressed` vs tag-only ŌĆö only 5/15 phases comply with canonical key." A backfilled SUMMARY that uses `requirements_addressed` (Phase 14's drift) or tag-only (Phase 15's drift) perpetuates the problem.

**How to avoid:** Every backfilled SUMMARY MUST use `requirements_completed: [REQ-ID, ...]` in YAML frontmatter. This is the canonical key per the audit. Mention this in the plan's must_haves checklist explicitly.

### Pitfall 6: Phase 5 ROADMAP ┬¦66-┬¦87 plan-list checkbox staleness vs disk reality

**What goes wrong:** ROADMAP shows 05-07 as `[x]` and 05-01..06 + 05-08..09 as `[ ]`. But the disk shows SUMMARYs for 05-02..08 (7 SUMMARYs). The disk is correct; ROADMAP is stale. A plan that tries to "reconcile" these by deleting SUMMARYs would lose information.

**How to avoid:** The 15.2-06 traceability plan should flip ROADMAP checkboxes to match disk reality (`[x]` for every plan that has a SUMMARY on disk; mention 05-09 status explicitly based on whether it backfilled or not). Don't treat ROADMAP as authoritative for this ŌĆö the SUMMARY files are the source of truth, and the v33.3 audit is the authoritative reconciliation.

### Pitfall 7: 09-01-SUMMARY.md scope leak

**What goes wrong:** A planner might be tempted to backfill 09-01-SUMMARY in 15.2 since it's just one missing file. Per ROADMAP ┬¦239 / ┬¦247 this is **explicitly out of scope** ŌĆö 09-01-SUMMARY waits for the canonical SVG that Phase 15.3 produces. Backfilling it from existing PNG-only state would lock in incorrect content that 15.3 then has to overwrite.

**How to avoid:** 15.2-05 (Phase 9 backfill plan) ships **09-VERIFICATION.md only**. The plan must_haves should explicitly call out "09-01-SUMMARY.md NOT produced ŌĆö deferred to Phase 15.3."

### Pitfall 8: Nyquist back-fill scope creep

**What goes wrong:** While reviewing missing artifacts, a planner might notice "Phase 5/6/7/8/9 also have no VALIDATION.md" and add Nyquist back-fill to 15.2's scope. Per ROADMAP ┬¦330 this is **explicitly deferred**: "Phase 15.5 Nyquist back-fill DEFERRED ŌĆö accepting Nyquist tooling tech-debt for v1; framework will apply going forward only."

**How to avoid:** The 15.2 plans must explicitly state "VALIDATION.md authoring NOT in scope per ROADMAP ┬¦330" in their must_haves block. The plan-checker should reject any 15.2 plan that creates `*-VALIDATION.md` files.

### Pitfall 9: REQUIREMENTS.md inline checkbox vs traceability table double-update

**What goes wrong:** REQUIREMENTS.md has TWO independent representations of REQ-ID status: (a) inline checkboxes in ┬¦11-┬¦95 (e.g., line 36 `- [ ] **AUTH-01**: ...`), and (b) the Traceability table at ┬¦229-┬¦338. The audit's "67 [x] / 39 [ ]" tally at line 424 refers to the *inline checkboxes*, not the table. Flipping only the table leaves the inline checkboxes stale; flipping only the inline checkboxes leaves the table stale.

**How to avoid:** Plan 15.2-06 must update **both** the inline checkboxes AND the table rows in the same commit. Use grep to find all stale `- [ ] **AUTH-` etc., then verify both representations match after the edit.

### Pitfall 10: SUMMARY commit hash forensics may exceed budget

**What goes wrong:** Some backfilled SUMMARYs (especially Phase 8's 5 of-them) need commit hash references for "what landed" sections. Finding those commit hashes via `git log --grep "08-01"` is tractable, but `git log -- js/app.js` for the relevant date range will return hundreds of commits. A naive approach (read each one) will burn token budget fast.

**How to avoid:** The plan should pre-document the commit-finding strategy:
1. Use `git log --oneline --all --grep "08-0N\|INTENT-0M\|D-XX" -- .` for fast subject-line grep
2. Cross-reference plan completion dates from directory mtimes (Phase 8 PLANs are dated 2026-04-21; SUMMARYs should bracket 2026-04-21..22)
3. If exact hashes can't be found, cite the production-deployed state ("commits between 2026-04-21 and 2026-04-22; precise hashes lost to retroactive backfill") ŌĆö the deploy is what matters.
4. **Acceptable degradation:** SUMMARYs without exact commit hashes are still better than no SUMMARYs. The verification trail is restored even if commit-level forensics is approximate.
