<objective>
Plan 02 — Per-title parent override surface on the detail modal (Wave 2,
sequential because js/app.js was already modified by Plan 19-01).

When a kid-mode-blocked title's detail modal is opened by a parent user,
render a "Show this anyway for tonight" link in the detail body. Tapping
adds the titleId to `state.kidModeOverrides` (the Set Plan 19-01 introduced),
re-renders Tonight + Library, closes the detail modal, and writes a Sentry
breadcrumb for product-analytics ("kids excluded but parents watch anyway"
frequency signal).

Three render conditions ALL must hold (per D-10..D-12):
1. `state.kidMode === true` — kid-mode is currently active
2. `tierFor(t.rating) > 2` — title is currently blocked by the kid-mode cap
3. `isCurrentUserParent() === true` — current user has parent privileges

Override behavior is session-scoped (D-13): the override Set is already
cleared by Plan 19-01's toggleKidMode on every transition. Multiple titles
can be overridden in a single session; each override is per-title.

This plan is small (~30 LOC across one js/app.js splice + one window-handler
+ one CSS rule). Wave 2 — sequential to Plan 19-01 only because both edit
js/app.js (Phase 14/15.5 same-file-overlap precedent → next-wave dependency).

Purpose: deliver the parent-only escape hatch. Closes the "but I'm a parent
and I want to watch this R-rated movie myself after the kids go to bed"
edge case while preserving the core safety contract (kids on the couch see
no R/PG-13).

Output: `js/app.js` extended with override link rendering inside
renderDetailShell + window.kidModeOverrideTitle handler; `css/app.css`
extended with `.kid-mode-override-link` styling.

Step 3 — Add the `window.kidModeOverrideTitle` handler near other detail-modal
window handlers (search for `window.addSimilar` at js/app.js:7630 and add
the new handler immediately AFTER that function, in the same handler cluster):

```javascript
// Phase 19 / D-11 — parent override: bypass kid-mode cap for this title in this
// session. Adds titleId to state.kidModeOverrides Set, re-renders affected
// surfaces, closes the detail modal, writes Sentry breadcrumb for analytics.
window.kidModeOverrideTitle = function(titleId) {
  if (!titleId) return;
  if (!isCurrentUserParent()) {
    // Defense in depth — render-side gate already hides the link for non-parents,
    // but a non-parent invoking the handler via DevTools should still be denied.
    flashToast('Only parents can do this', { kind: 'warn' });
    return;
  }
  if (!state.kidModeOverrides) state.kidModeOverrides = new Set();
  state.kidModeOverrides.add(titleId);
  // Sentry breadcrumb — Claude's discretion per CONTEXT.md, surfaces frequency of
  // "kids excluded but parents watch anyway" so we can re-evaluate the cap if needed.
  try {
    if (typeof Sentry !== 'undefined' && Sentry.addBreadcrumb) {
      const t = (state.titles || []).find(x => x.id === titleId);
      Sentry.addBreadcrumb({
        category: 'kid-mode-override',
        message: `parent override applied for ${titleId}`,
        level: 'info',
        data: {
          titleId,
          rating: t ? t.rating : null,
          by: state.me ? state.me.id : null
        }
      });
    }
  } catch (e) { /* never let analytics block UX */ }
  flashToast('Showing this for tonight', { kind: 'info' });
  // Close the detail modal so the re-rendered Tonight tab is visible.
  if (typeof closeDetailModal === 'function') closeDetailModal();
  if (typeof renderTonight === 'function') renderTonight();
  if (typeof renderLibrary === 'function') renderLibrary();
};
```
