# Phase 15.1 Plan 01: Wave 1 Client-Side Prep Summary

**One-liner:** Lands all client-side prep work that the Wave 2 rule tightening will depend on — converts three `t.progress` writers (writeMemberProgress + clearMemberProgress + Trakt-sync ingest) from full-map merge to dotted-path single-inner-key shape, stamps `actingTupleKey` on `setTupleName` + `cv15CoWatchPromptDecline` to enable the family-doc 5th-branch participant regex, and gates the `.cv15-tuple-rename` pencil glyph on participation while surfacing `setBy` attribution beside named tuples — all under the locked HOSTING FIRST → RULES SECOND ordering so prod writers conform before Wave 2 rules tighten.

## What Shipped

### Task 1 — `t.progress` writers converted to dotted-path single-inner-key shape

Three writer callsites in `js/app.js` switched from full-map merge to dotted-path so the Wave 2 4th sub-rule's `affectedKeys().hasOnly([memberId])` check will pass:

**`writeMemberProgress` (js/app.js:8414-8434):**

Before:
```js
const prevProgress = t.progress && typeof t.progress === 'object' ? { ...t.progress } : {};
prevProgress[memberId] = { season, episode, updatedAt: Date.now() };
await updateDoc(doc(titlesRef(), titleId), { ...writeAttribution(), progress: prevProgress });
```

After:
```js
await updateDoc(doc(titlesRef(), titleId), {
  ...writeAttribution(),
  [`progress.${memberId}`]: { season, episode, updatedAt: Date.now() }
});
```

The Trakt-push tail (lines 8434-8440) is preserved verbatim.

**`clearMemberProgress` (js/app.js:8438-8455):**

Before: full-map delete via `delete prevProgress[memberId]` + `progress: prevProgress`.

After: `[`progress.${memberId}`]: deleteField()` (deleteField already imported at js/app.js:1).

**Trakt-sync ingest path (js/app.js:749-771):**

Before: `const update = { progress: prevProgress }` after rebuilding the full progress map.

After: `const update = { [`progress.${meId}`]: { season, episode, lastWatchedAt, updatedAt, source: 'trakt' } }` — only the actor's slice. The CREATE branch immediately below (js/app.js:771-786) is unmodified — CREATE remains permissive per RESEARCH §Q1.

The `update[`ratings.${meId}`]` line stayed unchanged (already dotted-path).

### Task 2 — `actingTupleKey` stamped on family-doc tupleKey-keyed writers

**`setTupleName` (js/app.js:8552-8569):** added `actingTupleKey: tupleKeyStr` between the dotted-path tupleNames slot and the `writeAttribution()` spread, mirroring the `writeTupleProgress` pattern at js/app.js:8488.

**`cv15CoWatchPromptDecline` (js/app.js:1063-1071):** added `actingTupleKey: c.tupleKey` to the existing dotted-path coWatchPromptDeclined updateDoc payload. The existing `isSafeTupleKey(c.tupleKey)` guard, the optimistic local update, and the catch block with Sentry breadcrumb under category `coWatchPromptDeclined` all preserved unmodified.

These stamps enable Wave 2's family-doc 5th-branch participant regex `request.resource.data.actingTupleKey.matches('(^|.*,)' + memberId + '(,.*|$)')`.

### Task 3 — Pencil-glyph participant gate + setBy attribution surface

**`renderCv15TupleProgressSection` (js/app.js:8870-8902):**

- Added `const meId = (state.me && state.me.id) || ''` and `const isParticipant = meId && tk.split(',').includes(meId)` inside the row mapper.
- `renameBtn` now renders only when `isParticipant === true`; non-participants get an empty-string `renameBtn` so no pencil ever appears in their DOM.
- Added a `setByMarkup` block that, when `customName !== null` and `state.family.tupleNames[tk].setBy` is set and resolves to a member name, renders `<span class="cv15-tuple-setby" title="Renamed by {name}">· Renamed by {name}</span>` beside the rename button.
- Both `setByMember` lookup (`state.members.find(m => m && m.id === setBy)`) and `escapeHtml` of the resolved name are defensive against missing data.

**`cv15ShowRenameInput` (js/app.js:8956-8964):** added a hard guard at the top of the function — `const meId = (state.me && state.me.id) || ''; if (!meId || !tupleKeyStr.split(',').includes(meId)) return;` — so even if a non-participant somehow triggers the delegated click handler (DOM injection / dev console / out-of-date PWA), the rename input never opens.

The rest of `cv15ShowRenameInput` (input creation, addEventListener for keydown/blur, span.replaceWith, focus + select) is unmodified.

## Lockstep-deploy footnote

`actingTupleKey`-stamped writes (Task 2) will be **REJECTED by the current prod 5th-branch allowlist** until Plan 15.1-02 ships its allowlist extension. The current `firestore.rules` 5th UPDATE branch reads:

```
.hasOnly(['tupleNames', 'coWatchPromptDeclined', 'actingUid', 'managedMemberId', 'memberId', 'memberName'])
```

— `actingTupleKey` is not in that allowlist. The locked decision per CONTEXT §Cross-cutting is to ship the stamp unconditionally because:

- (a) `setTupleName` is rare (~0-1 writes/family/day per RESEARCH §Q4)
- (b) `cv15CoWatchPromptDecline` has a silent-failure catch path with Sentry breadcrumb under `category='coWatchPromptDeclined'`, so prod failures during the deploy window are observable
- (c) The Wave 1 → Wave 2 deploy gap is ≤1 day (one hosting-deploy cycle, NOT a 1-2 week PWA cache roll)

**Plan 15.1-03 encodes the deploy ordering** (HOSTING FIRST → RULES SECOND, except for the `actingTupleKey` allowlist extension which the user may choose to deploy lockstep).

Tasks 1 and 3 are 100% backward-compatible with the current rules:
- Task 1's dotted-path writes are accepted by the current permissive title-doc UPDATE rule (no `affectedKeys()` constraint exists yet — that's Wave 2)
- Task 3 is render-only — no writer, no rule impact

So if Plan 15.1-02 slips, Tasks 1 and 3 can ship to prod immediately while Task 2 awaits the rules deploy. The plan author (and the user) chose to ship all three together to keep the change atomic.
