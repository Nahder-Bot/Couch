## Implementation Decisions

The todo + audit artifacts already lock the implementation patterns. Plans MUST honor these locked decisions; deviations require a /gsd-discuss-phase round-trip.

### F-OPEN-01: Per-member isolation on `t.progress` map (HIGH)

**Locked.** Mirror the existing `mutedShows` rule pattern:
- **Rules side.** Title-doc UPDATE branch (`firestore.rules:366-432`) gets a 4th sub-rule restricting writes that touch `progress` to a dotted-path diff whose affected inner keys are `hasOnly([memberId])` (or `[managedMemberId]` when proxying). The existing 3 sub-rules for `tupleProgress` / `mutedShows` / `liveReleaseFiredFor` are the pattern reference.
- **Writer side.** `writeMemberProgress` (`js/app.js:8407`) refactors from full-map writes (`{ progress: prevProgress }`) to dotted-path writes (`{ [`progress.${meId}`]: ... }`) so the rule's `affectedKeys()` exposes a single inner key. Any other Trakt-sync ingest paths or post-watchparty progress writers must follow the same pattern — the planner is responsible for finding all callsites.
- **Trust side.** Once `progress[X].source` cannot be forged by other members, the Trakt overlap detector at `js/app.js:951-994` regains its trust assumption — no client-side detector changes required.

### WR-02: `memberId` regex anchor + `auth.uid` cross-validation (HIGH)

**Locked.** Two-layer fix per the 15-REVIEW.md WR-02 §Fix block:
- **Anchor.** Add `request.resource.data.memberId is string && request.resource.data.memberId.matches('^m_[A-Za-z0-9_-]+$')` as a precondition to every rule branch that consumes `memberId` (title-doc UPDATE — `tupleProgress` / `mutedShows` sub-rules; family-doc 5th UPDATE branch). Same anchor applied to `managedMemberId` when present. Defangs RE2 meta-character injection in the title-doc `actingTupleKey` runtime regex.
- **Cross-validate.** `validAttribution()` (`firestore.rules:59-66`) extended to verify `request.resource.data.memberId == request.auth.uid` (or `managedMemberId(uid)` for proxy-acted writes via `get(/databases/$(database)/documents/families/$(familyCode)/members/$(managedMemberId)).data.managedBy == uid()`). Closes the spoof path that lets a member set `memberId: 'm_VICTIM'` while keeping their own `actingUid`.
- **Scope.** Apply the anchor across ALL rule branches that consume `memberId` — title-doc isolation, `mutedShows.hasOnly([memberId])`, family-doc 5th UPDATE allowlist. The audit explicitly notes the same payload-trust pattern recurs across `couchSeating`, `couchInTonight`, `tupleNames` — but the broader Phase-5 backfill is OUT of scope (see Phase Boundary). For Phase 15.1 we anchor the Phase-15-introduced surfaces; planner may add anchors to other Phase-15-touched branches if doing so doesn't widen the blast radius.

### F-OPEN-02: `tupleNames` participant check (MEDIUM)

**Locked — BOTH layers per the todo §3 §"Recommend BOTH":**
- **Server-side (SEC-15-1-03).** Family-doc 5th UPDATE branch (`firestore.rules:181-200`) extended to peek inside `tupleNames.<tk>` affected keys and verify `actingMemberId in tk.split(',')` for each tupleKey being written. Mirrors the title-doc HIGH-1 `actingTupleKey` echo pattern. Requires `setTupleName` (`js/app.js:8517`) to stamp `actingTupleKey: tupleKeyStr` on its dotted-path write. Same enforcement extended to `coWatchPromptDeclined.<tk>` (the only other tupleKey-keyed map under that branch).
- **Client-side (SEC-15-1-05).** Two UI changes:
  1. Gate the `.cv15-tuple-rename` pencil glyph in `cv15ShowRenameInput` + `renderCv15TupleProgressSection` so it only renders when `state.me.id ∈ tk.split(',')`. ~10 lines client-only. Closes the casual-griefing path even before the rule lands; primary value is reducing UI affordance.
  2. Surface the existing `setBy` field as a subtle "Renamed by {memberName}" attribution beside the tuple name on hover/tap — turns silent vandalism into attributable action.

Both ship together; the rule is the actual security boundary, the client changes are defense-in-depth + UX.

### WR-03: Rules tests #33-#35 (MEDIUM)

**Locked.** Three tests in `tests/rules.test.js`, mirroring the file's existing #23-#32 Phase-15 isolation tests:
- **#33** — authed member writes own `coWatchPromptDeclined` tuple → ALLOWED. Pins the cross-session decline-persistence path that 15-08's Task 4 enabled.
- **#34** — authed member writes `coWatchPromptDeclined` for a tuple they're NOT in → DENIED. Gated on SEC-15-1-03 landing; will fail until then. The plan-checker should expect this test to be reordered if SEC-15-1-03 ships in a later wave.
- **#35** — forged `memberId: '.*'` writing `tupleProgress` → DENIED. Gated on SEC-15-1-02 landing.

Plus regression tests for any rule branches whose behavior is changed by SEC-15-1-01..03 (the planner identifies these from the touched-rule diff). All tests load `firestore.rules` from the source repo (the post-WR-01 source file is now in sync with the deploy mirror, so `tests/rules.test.js:22` `RULES_FILE` resolution is correct).

### Cross-cutting

- **No `sw.js` CACHE bump in this phase** unless a user-visible UI change ships (SEC-15-1-05's pencil-glyph hide IS user-visible — bump required). Bump value: `couch-v35.1-security-hardening`. Per CLAUDE.md: auto-bumped via `bash scripts/deploy.sh 35.1-security-hardening`.
- **Cross-repo deploy ordering — REVERSED from Phase 15.** Phase 15 added new fields, so rules had to deploy FIRST to permit them. Phase 15.1 TIGHTENS existing fields, so the safe ordering is HOSTING FIRST → RULES SECOND: deploy hosting (`bash scripts/deploy.sh 35.1-security-hardening`) which lands the dotted-path writers + actingTupleKey stamps + UI changes, THEN deploy rules (`firebase deploy --only firestore:rules --project queuenight-84044` from `~/queuenight`). Rationale: deploying tightened rules BEFORE writers update would break in-flight prod writes (especially SEC-15-1-01's multi-key `progress` writes). RESEARCH.md §Q6 has the full analysis. Note: there is a deliberately-accepted gap window for SEC-15-1-03's `actingTupleKey` writes (writers stamp it before the family-doc 5th-branch allowlist accepts it), mitigated by low write frequency + silent-failure catch on `cv15CoWatchPromptDecline` + Sentry breadcrumbs surfacing failures. The user may also choose lockstep (deploy hosting + rules within the same minute) — Plan 15.1-03 Task 2 documents both options and the SEC-15-1-01 vs SEC-15-1-03 ordering tradeoff.
- **Source/mirror parity for `firestore.rules`.** Phase 15 close-out fixed the WR-01 drift and brought source in sync. Phase 15.1 rule edits MUST land in BOTH `couch/firestore.rules` (source — what tests load) AND `~/queuenight/firestore.rules` (deploy mirror) in the same commit pair. RUNBOOK §H or a CI diff check should already cover this; planner should re-verify the deploy.sh path doesn't auto-mirror rules (it does NOT per WR-01's analysis).

## Specific Ideas

Concrete details from the audit + review that plans should encode verbatim:

### Rule code shapes the planner should mirror

For SEC-15-1-01 (mirroring the existing `mutedShows` pattern at `firestore.rules:415-425`):
```
&& (
  !affectedKeys().hasAny(['progress'])
  || (
    request.resource.data.progress.diff(resource.data.progress).affectedKeys()
      .hasOnly([memberIdOrManagedMemberId])
  )
)
```

For SEC-15-1-02 (precondition guard):
```
&& request.resource.data.memberId is string
&& request.resource.data.memberId.matches('^m_[A-Za-z0-9_-]+$')
&& (
  !('managedMemberId' in request.resource.data)
  || (
    request.resource.data.managedMemberId is string
    && request.resource.data.managedMemberId.matches('^m_[A-Za-z0-9_-]+$')
  )
)
```

For SEC-15-1-03 (mirroring the title-doc `actingTupleKey` echo+regex pattern at `firestore.rules:397-404`, but applied to family-doc tupleNames + coWatchPromptDeclined):
```
&& request.resource.data.actingTupleKey
    .matches('(^|.*,)' + request.resource.data.memberId + '(,.*|$)')
&& request.resource.data.tupleNames.diff(resource.data.tupleNames).affectedKeys()
    .all(tk, tk == request.resource.data.actingTupleKey)
```

### Writer code shape (SEC-15-1-01 dotted-path conversion)
```js
// js/app.js writeMemberProgress (~line 8407) — BEFORE
await updateDoc(doc(titlesRef(), titleId), { ...writeAttribution(), progress: prevProgress });

// AFTER — dotted-path single-inner-key write
await updateDoc(doc(titlesRef(), titleId), {
  ...writeAttribution(),
  [`progress.${meId}`]: { season, episode, lastWatchedAt: Date.now(), source: 'manual' }
});
```
Planner identifies ALL callsites (manual rate flow, post-watchparty progress, Trakt-sync ingest) and converts them in one plan to avoid mid-conversion rule failures.

### Test code shapes (mirror tests #23-#32 style at `tests/rules.test.js`)
- #33 ALLOWED — `member.doc('families/fam1').set({ coWatchPromptDeclined: { 'm_UID_MEMBER,m_UID_OWNER': 1234567890 }, actingUid: UID_MEMBER, memberId: 'm_UID_MEMBER', actingTupleKey: 'm_UID_MEMBER,m_UID_OWNER', memberName: 'Member' }, { merge: true });` → assertSucceeds
- #34 DENIED — same payload but tuple `'m_UID_OUTSIDER,m_UID_OTHER'` (current member not in tuple) → assertFails
- #35 DENIED — `memberId: '.*'` with otherwise-valid tupleProgress payload → assertFails (regex anchor catches it)

### Client-side code shape (SEC-15-1-05 pencil-hide)
