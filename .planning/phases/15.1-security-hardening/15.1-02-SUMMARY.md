## What Shipped

### Task 1 — SEC-15-1-02 anchor + SEC-15-1-01 progress sub-rule

**`firestore.rules` validAttribution() (lines 55-82):**

Before: 8 lines requiring `actingUid == uid()` + optional managedMemberId proxy proof.

After: 28 lines extending with:
- `request.resource.data.memberId is string` precondition
- `request.resource.data.memberId.matches('^m_[A-Za-z0-9_-]+$')` anchor
- managedMemberId branch wrapped to also check `is string` + the same anchored regex BEFORE the existing `members/{managedMemberId}.managedBy == uid()` proof.

The character class `[A-Za-z0-9_-]` places `-` last so it's a literal, not a range. Both `^` and `$` anchors are mandatory because RE2 partial-matches by default — without anchors, `m_VICTIM.*` would match.

**`firestore.rules` title-doc UPDATE rule (lines 450-471):**

Appended a 4th sub-rule mirroring the mutedShows pattern at lines 430-449. When the diff touches `progress`:
- `request.resource.data.progress is map` type guard
- Property-existence guards on both sides of the `.diff()`: `'progress' in request.resource.data ? ... : {}` AND `'progress' in resource.data ? ... : {}` (mandatory — bare access errors when the doc has no progress field yet)
- `affectedKeys().hasOnly([memberId-or-managedMemberId])` constrains the single touched inner key to the actor's surface

The mirror `~/queuenight/firestore.rules` received the IDENTICAL edit. Pre-existing drift was reconciled inline (the source had a stray blank `//` comment line at the old line 191 that the mirror lacked — removed for byte-identity).

**`tests/rules.test.js` tests #1 + #2 backfilled:** the SEC-15-1-02 anchor now requires `memberId` for ALL attributedWrite paths. Tests #1 and #2 had minimal payloads omitting it, which the live `writeAttribution()` from `js/utils.js:92` would never produce. Added `memberId: 'm_UID_MEMBER'` + `memberName: 'Member'` (and the equivalent for owner) to match the real writer contract.

### Task 2 — SEC-15-1-03 family-doc 5th UPDATE branch participant check

**`firestore.rules` family-doc UPDATE 5th branch (lines 197-262):**

Replaced the entire 5th branch (was 20 lines including the leading comment block; now 66 lines) with the participant-checked version. The five-part structure:

1. **(a) attribution baseline** — `attributedWrite(familyCode)` (which now includes the SEC-15-1-02 anchor)
2. **(b) outer allowlist extended** — `actingTupleKey` added to the 6 prior fields → 7 total: `'tupleNames', 'coWatchPromptDeclined', 'actingTupleKey', 'actingUid', 'managedMemberId', 'memberId', 'memberName'`
3. **(c) actingTupleKey echo + comma-boundary regex** — only fires when the diff touches tupleNames or coWatchPromptDeclined. Two-layer regex:
   - Self-sanitization: `actingTupleKey.matches('^[A-Za-z0-9_,-]+$')` (defends against RE2 meta injection in the inner regex)
   - Membership: `actingTupleKey.matches('(^|.*,)' + memberId + '(,.*|$)')` (the actor must appear at a comma boundary)
4. **(d) tupleNames inner-key constraint** — `request.resource.data.tupleNames.diff(resource.data.tupleNames).affectedKeys().hasOnly([request.resource.data.actingTupleKey])` with mandatory property-existence guards.
5. **(e) coWatchPromptDeclined inner-key constraint** — same pattern as (d) for coWatchPromptDeclined.

The mirror received the IDENTICAL edit. Byte-identity preserved post-Task-2 (`diff` exit 0).
