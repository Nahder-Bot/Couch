## Summary

This is a tight hardening pass — the audit + review already enumerated the bugs, exploit paths, and recommended fixes. Research focused on the seven implementation-detail questions that the planner needs answered before splitting work into atomic plans. Five of the seven questions resolve cleanly against artifacts already in this repo (Phase 15-01-SUMMARY.md is unusually rich — it documents every rules-language deviation encountered when 15-01 was executed against the 2026-04-27 Firestore emulator). The remaining two (rules-side `affectedKeys()` granularity for dotted-path writes, and `matches()` anchor behavior) needed careful primary-source verification because the MDN-style external docs are sparse for the rules CEL dialect.

**The three load-bearing findings the planner MUST encode:**

1. **`memberId != auth.uid`.** `state.me.id` is generated as `'m_' + Date.now() + '_' + Math.random()` (`js/app.js:3418`), NOT as `'m_' + auth.uid`. The 15-01 executor already discovered this and corrected the plan's narrative — see Plan 15-01-SUMMARY.md "Auto-fixed Issues #1." The `members/{memberId}.uid == auth.uid` doc is the only authoritative link. Any rule that compares "actor identity" must either use `request.resource.data.memberId` (with `managedMemberId` proxy fallback — already the convention) **or** do a `get()` on the member doc. The "extend `validAttribution()` with `auth.uid == memberId`" recommendation in 15-REVIEW.md WR-02 §Fix is **wrong as stated**; it would deny every existing legit write.

2. **Dotted-path writer + `<map>.diff(<map>).affectedKeys()` is the working pattern.** Phase 15 already proved this in production: `setTupleName` at `js/app.js:8546` does `updateDoc(..., { [`tupleNames.${tk}`]: slot, ... })` and `coWatchPromptDeclined` at `js/app.js:1064` does the same. The rule side reads `request.resource.data.tupleNames` as a full map and then `.diff()`s it against `resource.data.tupleNames` — which gives an inner-key set. The catch is that `resource.data.<map>` errors when the field is absent on the existing doc (FIRST write); the working idiom (also from 15-01-SUMMARY auto-fix #3) is `('tupleNames' in resource.data ? resource.data.tupleNames : {})`.

3. **`request.resource.data.tupleProgress is map` works after a dotted-path inner-key write.** This is the non-obvious win: even though the writer only sends `{[`progress.${meId}`]: ...}`, the `request.resource.data.progress` view exposes the resulting full map (existing keys merged with the inner-key write), and `.diff()` gives ONE inner key. This is exactly how the existing `tupleProgress` HIGH-1 rule (firestore.rules:392-412) is structured. SEC-15-1-01 just clones that pattern.

**Primary recommendation:** Plan 15-01-SUMMARY.md is the single most important reference for this phase. Every rule deviation, idiom, and emulator-tested pattern is documented there. The planner should read it FIRST when shaping each rule fix.

## Q1. `affectedKeys().diff()` semantics for nested maps under dotted-path writes

**Answer (HIGH confidence — verified against existing firestore.rules + 15-01-SUMMARY auto-fixes):**

When the writer issues `updateDoc(docRef, { [`progress.${meId}`]: payload })`:

- **`request.resource.data.progress`** evaluates to the FULL post-merge map (existing inner keys + the inner key being written). It is a `rules.Map`, not null. [VERIFIED: Plan 15-01-SUMMARY auto-fix #3 + the existing tupleProgress rule at `firestore.rules:392-412` which depends on this behavior]
- **`resource.data.progress`** evaluates to the existing pre-write map — but this errors with `Property progress is undefined on object` if the existing doc has NO `progress` field yet (first write). [VERIFIED: 15-01-SUMMARY Auto-fixed Issue #3]
- **`request.resource.data.progress.diff(resource.data.progress).affectedKeys()`** returns a `rules.Set` of the CHANGED INNER KEYS. For a single dotted-path write `{[`progress.${meId}`]: ...}`, this set has size 1 and contains exactly `meId`. [VERIFIED: tupleProgress rule body at firestore.rules:392-412 ships in production with this exact pattern + tests #27/#29 pass]
- **`hasOnly([memberId])`** on the result correctly enforces that exactly one inner key is being touched and it equals the actor's id. [VERIFIED: existing mutedShows rule at firestore.rules:415-432 + tests #28/#30]

**The required idiom (working pattern from 15-01-SUMMARY):**

```
!request.resource.data.diff(resource.data).affectedKeys().hasAny(['progress'])
|| (
  request.resource.data.progress is map
  && ('progress' in request.resource.data ? request.resource.data.progress : {})
       .diff('progress' in resource.data ? resource.data.progress : {})
       .affectedKeys()
       .hasOnly([
         ('managedMemberId' in request.resource.data && request.resource.data.managedMemberId is string)
           ? request.resource.data.managedMemberId
           : request.resource.data.memberId
       ])
)
```

The two property-existence ternaries (`'X' in resource.data ? ... : {}`) are MANDATORY — bare `resource.data.progress` errors when the property is absent. This was Auto-fix #3 in Plan 15-01.

**What happens when the ENTIRE map is rewritten (legacy `progress: {...}` write before refactor lands):** The diff explodes — `affectedKeys()` returns the union of all inner keys that differ between old and new. A write that touches more than one inner key correctly fails `hasOnly([memberId])`. This means **rules can deploy slightly AFTER writer if needed** (rule denies legacy writer, but writer still works against pre-rule prod). It also means the writer + rule should ship in lockstep — see §Q6.

**Implementation implication for the planner:**
- For SEC-15-1-01: this is exactly the existing `mutedShows` rule template; clone-as-sibling.
- For SEC-15-1-03's `tupleNames` and `coWatchPromptDeclined`: these already use dotted-path writes on the FAMILY doc (`families/{code}.tupleNames.${tk}` at `js/app.js:8547` and `families/{code}.coWatchPromptDeclined.${tk}` at `js/app.js:1064`), so the writer side is DONE. Only the rule needs adding. This is the cleanest plan to ship.
- For SEC-15-1-01's `t.progress` writer: there are TWO callsites — the canonical `writeMemberProgress` at `js/app.js:8412-8429` AND the Trakt-sync ingest path at `js/app.js:760` (which does `update = { progress: prevProgress }`). BOTH must convert. There is also a CREATE path at `js/app.js:774` that builds `progress: {[meId]: ...}` inline as part of `createTitleFromTrakt` — this is fine (CREATE doesn't trigger UPDATE rule and the create rule remains permissive `attributedWrite`).

**Landmines for `<read_first>` blocks:**
- ⚠ `resource.data.progress` errors when absent — always guard with `'progress' in resource.data ? ... : {}`. (Plan 15-01-SUMMARY Auto-fix #3.)
- ⚠ `clearMemberProgress` at `js/app.js:8432-8441` does `delete prevProgress[memberId]` then writes the FULL map. After conversion, this needs to use `deleteField()` via dotted-path: `{[`progress.${memberId}`]: deleteField(), ...}` — there's a precedent for this in `writeMutedShow`'s unmute path (per 15-02-SUMMARY).
- ⚠ The `progress` map has a documented LEGACY shape: a flat `progressSeason`/`progressEpisode` pair on the title doc itself (`js/app.js:8404-8407` references both). The dotted-path conversion does NOT touch those legacy fields, so no breakage there.

## Q2. RE2 regex behavior in `matches()` — anchors + meta-characters

**Answer (HIGH confidence on anchor behavior, MEDIUM confidence on partial-match semantics with attribution):**

Firestore Security Rules use Google's RE2 regex engine. [CITED: Firebase docs note rules use RE2 syntax — see Sources] [VERIFIED: tested in Phase 15-01 against the 2026-04-27 Firestore emulator]

- **`matches()` performs PARTIAL match by default.** [CITED: `https://swtch.com/~rsc/regexp/regexp3.html` — RE2 PartialMatch returns true if regex matches part of subject] This is why the audit's reported attack works — `actingTupleKey.matches('(^|.*,).*(,.*|$)')` matches ANY string containing a comma OR ANY string at all, depending on what `memberId` injects.
- **To anchor a full-string match, you MUST add `^` and `$` explicitly.** [CITED: RE2 docs at `https://github.com/google/re2/wiki/Syntax`]
- **Therefore SEC-15-1-02's anchor regex MUST be `'^m_[A-Za-z0-9_-]+$'`** — without the anchors, `memberId.matches('m_[A-Za-z0-9_-]+')` would still admit `'m_aaaa.*'` (the engine finds the prefix `m_aaaa` and returns true, ignoring the `.*` tail).
- **Inside an anchored character class, `_` and `-` are SAFE** (not RE2 meta-characters). [CITED: RE2 syntax docs] `-` only acts as a range operator inside `[...]`; placing it last (`[A-Za-z0-9_-]`) is the canonical safe form. The current `isSafeTupleKey` regex at `js/app.js:8268` (`/^[A-Za-z0-9_,-]+$/`) uses the same idiom and has shipped without issue.

**The runtime regex injection in the existing title-doc `actingTupleKey` rule:**

The current rule at `firestore.rules:404-411` constructs:
```
'(^|.*,)' + memberId + '(,.*|$)'
```
With `memberId = '.*'`, the constructed regex becomes `(^|.*,).*(,.*|$)`. Even though this regex looks bounded, against the typical `actingTupleKey` `'m_X,m_Y'` it matches because `(^|.*,)` matches the empty prefix at position 0, the middle `.*` matches everything, and `(,.*|$)` matches the empty suffix at the end. **Every** `actingTupleKey` matches, including ones that exclude the actor.

**The fix locks both ends:** anchor `memberId` to `^m_[A-Za-z0-9_-]+$` BEFORE the regex is constructed. After that, the only injectable characters are `m_`, alphanumerics, `_`, and `-`, none of which are RE2 meta-characters. The RE2 engine then sees a literal substring that either is or isn't present in the actingTupleKey at the comma-boundary anchor positions.

**Implementation implication for the planner:**
- Anchor regex value: `'^m_[A-Za-z0-9_-]+$'` — exactly. NOT `'m_[A-Za-z0-9_-]+'`.
- Apply identically to `managedMemberId` when present (the proxy-acted path).
- The anchor predicate goes in `validAttribution()` if extending the helper, OR inline in each branch that consumes `memberId`/`managedMemberId`. The helper extension is cheaper (one place to maintain) and is what 15-REVIEW.md WR-02 §Fix recommends. The planner should default to extending `validAttribution()` unless rule-read-cost analysis (§Q4) flags an issue — and cost analysis says extending the helper is FREE because the regex check is local-only (no `get()`).
- After anchoring, the audit's regex-injection variant `memberId: 'm_[^,]*'` is also defanged — `[` is no longer in the allowed character class, so the precondition fails before the runtime regex is constructed.

**Landmines for `<read_first>` blocks:**
- ⚠ `matches()` partial-matches by default. ALWAYS use `^...$`.
- ⚠ Inside a character class, place `-` first or last (`[A-Za-z0-9_-]`) so it's a literal. Never `[A-Z-a-z]` (which is a range from `Z` to `a` and DOES NOT do what it looks like).
- ⚠ The existing `isSafeTupleKey` at `js/app.js:8268` is CLIENT-side validation only. It does not protect against an attacker bypassing the client and writing directly to Firestore via the SDK with a forged tupleKey containing a `.` — the rule must independently anchor.

## Q3. `memberId == auth.uid` is WRONG — the convention is `memberId = 'm_<random>'` linked via the member doc

**Answer (HIGH confidence — verified by reading `js/app.js:3418` + `js/utils.js:92-107` + Plan 15-01-SUMMARY Auto-fix #1):**

This is the most important finding in this research. The 15-REVIEW.md WR-02 §Fix block's recommendation to "extend `validAttribution()` to verify `request.resource.data.memberId == request.auth.uid`" **would deny every legitimate Phase 15 write in production**. Plan 15-01 already discovered this and patched it during execution.

**The actual data convention (verified from app.js):**

```js
// js/app.js:3418 — joinAsNew
const id = 'm_' + Date.now() + '_' + Math.random().toString(36).slice(2,8);
const member = { id, name, color, uid: state.auth ? state.auth.uid : null };
// state.me.id = id  ← memberId is set to this random string
```

```js
// js/app.js:3678 — createSubProfile
const memberId = 'm_' + Date.now() + '_' + Math.random().toString(36).slice(2,8);
const sub = { id: memberId, ..., managedBy: state.auth.uid /* NO uid field */ };
```

So:
- `memberId` is `'m_' + Date.now() + '_' + 6_random_chars` — a family-scoped opaque identifier.
- The link `memberId ↔ auth.uid` is via `members/{memberId}.uid == auth.uid` for adults, or `members/{memberId}.managedBy == auth.uid` for sub-profiles.
- `request.auth.uid` is the bare Firebase UID (e.g. `'KdL3...28chars'` from Firebase Auth) — has NO `m_` prefix.
- The test fixtures use `m_UID_OWNER`, `m_UID_MEMBER`, etc. — this is a **test convenience** that happens to make memberId look uid-derived, NOT the production convention.

**Three options for the SEC-15-1-02 cross-validation, ranked by tradeoffs:**

| Option | Mechanism | Cost | Strength | Notes |
|--------|-----------|------|----------|-------|
| **A. Trust `writeAttribution()` payload** | No new rule check — rely on the fact that `validAttribution()` already enforces `actingUid == auth.uid` and the writer always populates `memberId` from `state.me.id` | Free | WEAK — anchor alone (§Q2) is the only protection; an attacker crafting a direct SDK write CAN forge `memberId: 'm_VICTIM'` while keeping `actingUid: <self UID>` | This is the CURRENT (pre-15.1) state. Insufficient — that's the WR-02 attack. |
| **B. `get()` the member doc, verify `members/{memberId}.uid == auth.uid`** | Add to `validAttribution()`: `&& get(/databases/$(database)/documents/families/$(familyCode)/members/$(request.resource.data.memberId)).data.uid == uid()` (with managedBy fallback for sub-profiles) | +1 Firestore read per authed write | STRONG — mathematically correct; can't be forged without compromising auth | Rule reads are batched per-request (cached if same path is read twice in one rule eval). Firestore charges for rule `get()`s. See §Q4. |
| **C. Anchor + structural constraint** | `memberId.matches('^m_[A-Za-z0-9_-]+$')` (anchor, §Q2) + `'memberId' in request.resource.data` + maybe `actingUid is string` checks. NO uid-↔-memberId proof | Free | MODERATE — defangs the RE2 injection completely (which is the only attack the audit weaponized in WR-02). An attacker who knows another member's `memberId` string can still spoof memberId in writes — but post-anchor that only affects tupleProgress regex / mutedShows hasOnly checks, both of which then enforce per-key constraints via `actingTupleKey` echo (already in place) | This is what Plan 15-01 actually shipped (memberId comparison is the post-anchor source of truth for tuple/muted writes; identity-proof via `get()` is deferred). |

**Plan 15-01-SUMMARY's Auto-fix #1 chose Option C** — using `request.resource.data.memberId` (with `managedMemberId` proxy fallback) as the comparison value. This is what's already deployed for `tupleProgress` + `mutedShows`. SEC-15-1-02's job is to **add the anchor** so that the value being trusted is at least syntactically valid — not to add the `get()`-based identity proof (which is in Phase 15.1 CONTEXT.md's Deferred Ideas list).

**Implementation implication for the planner:**
- **Default plan:** Option C — extend `validAttribution()` to add the anchor only. No `get()`. This makes `validAttribution()`:
  ```
  function validAttribution(familyCode) {
    return isMemberOfFamily(familyCode)
      && request.resource.data.actingUid == uid()
      && request.resource.data.memberId is string
      && request.resource.data.memberId.matches('^m_[A-Za-z0-9_-]+$')
      && (
        !('managedMemberId' in request.resource.data)
        || (
          request.resource.data.managedMemberId is string
          && request.resource.data.managedMemberId.matches('^m_[A-Za-z0-9_-]+$')
          && get(/databases/$(database)/documents/families/$(familyCode)/members/$(request.resource.data.managedMemberId)).data.managedBy == uid()
        )
      );
  }
  ```
  The `managedMemberId` `get()` already exists in the helper (line 64); this just adds the anchor preconditions.
- **Decision question for the planner:** does anchoring break `legacyGraceWrite()` paths? Look at `firestore.rules:70-74`. `legacyGraceWrite()` requires `memberId is string` but doesn't require `actingUid`. The grace window is over (Phase 5 was a long time ago and the grace window was a one-time migration); but `legacyGraceWrite()` is still active in the helpers. **Recommendation:** scope the anchor change to `validAttribution()` (post-grace) ONLY, leaving `legacyGraceWrite()` untouched. If grace is somehow re-activated for another Phase, that's a Phase-5-hardening problem (already in CONTEXT.md Deferred Ideas).
- **`get()` extension:** explicitly DEFERRED per CONTEXT.md Deferred Ideas. Don't include unless rule-read cost analysis says it's free.

**Landmines for `<read_first>` blocks:**
- ⚠ Do NOT compare `memberId == auth.uid` directly — they have different shapes (`'m_<random>'` vs `'<firebase_uid>'`). 15-REVIEW.md's WR-02 §Fix has this wrong; the fix block in CONTEXT.md is also imprecise.
- ⚠ The existing helper's `managedMemberId` proof at `firestore.rules:64` is the canonical proxy-validation pattern — don't break it.
- ⚠ Tests use `m_UID_MEMBER`-shaped IDs which COINCIDENTALLY look like `'m_' + uid`. Don't read this as the production convention.

## Q4. Rule-read cost / quota considerations

**Answer (MEDIUM confidence — based on Firebase pricing docs + reasoning about Phase-15 write volumes):**

Each `get()` in a Firestore Security Rule counts as a billable Firestore document read at the standard rate ($0.06 per 100K reads on Blaze). [CITED: Firebase pricing — `https://firebase.google.com/pricing`] [VERIFIED: project memory confirms Couch is on Blaze]

**Per-request caching:** Firestore Rules cache identical `get()` calls within the SAME rule evaluation. So `validAttribution()` calling `get(.../members/$(managedMemberId))` once per write is one billable read, even if the rule references it twice. [CITED: Firebase rules behavior docs — `https://firebase.google.com/docs/rules/rules-behavior`]

**Phase-15 write volume estimate (rough, from 15-CONTEXT + summaries):**

| Surface | Write trigger | Estimated writes/family/day |
|---------|---------------|----------------------------|
| `writeMemberProgress` (manual) | User taps "I just watched S{N}E{M}" in detail modal | ~5-15 |
| `writeMemberProgress` (post-watchparty auto-track) | Watchparty ends → auto-track stash → user accepts | ~0-2 |
| `writeMemberProgress` (Trakt sync) | Periodic Trakt sync overlap detection | ~0-10 (only if Trakt connected) |
| `writeTupleProgress` | Tuple-keyed analog of above | ~0-5 |
| `setTupleName` | User renames a couch | ~0-1 (rare) |
| `writeMutedShow` | User toggles per-show kill switch | ~0-2 (rare) |
| `cv15CoWatchPromptDecline` | User declines a Trakt overlap prompt | ~0-1 (rare) |

**Worst-case: ~30 writes/family/day across ALL Phase-15 surfaces.** At 10K active families (commercial v1 target — far beyond current scale) that's 300K writes/day = ~9M/month.

**If we add a `get()`-based `members/{memberId}.uid == uid()` check** (Option B from §Q3) to `validAttribution()`: that's ~9M extra reads/month. Cost: **~$5.40/month at scale**. At current scale (single-digit families): pennies. Not material either way.

**The actual cost concern is latency, not money.** A rule `get()` on the member doc adds 5-20ms to every authed write's rule evaluation. For optimistic-UI surfaces this is invisible; for chained writes it adds up. Phase 15's surfaces are user-paced (a tap-and-wait pattern), so the latency is also non-issue.

**Implementation implication for the planner:**
- **Cost is not a reason to defer the `get()`-based identity proof.** If the planner wants to ship Option B (member-doc proof), the cost is negligible. The reason to defer is risk: (a) it changes `validAttribution()` semantics for ALL authed writes, including Phase 1-14 paths — risk of regression; (b) it requires a `members/{memberId}.uid` field to exist on every adult member doc, which is a schema assumption that needs verification; (c) it doesn't help sub-profile writes (which already use `managedMemberId` + `managedBy` proof at `firestore.rules:64`).
- **Recommendation:** ship Option C (anchor only, no `get()` extension) per CONTEXT.md's locked decision. Cost is not the gating concern. CONTEXT.md already lists the `get()` extension in Deferred Ideas with the rationale "Defer pending performance + cost analysis on the Firestore rules CALL count" — this research confirms cost isn't blocking, but CONTEXT.md still locks it as deferred (the planner cannot override).

**Landmines for `<read_first>` blocks:**
- ⚠ Don't add a rule `get()` that's already in `validAttribution()` — per-request caching makes this free, but it's a code-smell and an audit-trail hazard (someone reading the rule will think it's two reads).
- ⚠ If the planner DOES go beyond CONTEXT scope and add member-doc proof, ensure it's gated on `'memberId' in request.resource.data` first — `get()` on a missing path errors and denies the whole rule.

### Pattern 1: Dotted-path writer with `affectedKeys()` rule guard

**What:** Convert full-map writes to dotted-path writes so the rule can use `<map>.diff(<map>).affectedKeys().hasOnly([key])` for per-key isolation.

**Example (writer + rule pair):**


```
  || (
    request.resource.data.progress is map

         .diff('progress' in resource.data ? resource.data.progress : {})
```

### Pattern 2: `actingTupleKey` echo for tuple-keyed maps

**What:** Writer stamps `actingTupleKey: <tupleKey>` alongside the dotted-path tupleKey write. Rule asserts the diff hasOnly the echoed key AND the echoed key passes a regex containing the actor's memberId.

**When to use:** For maps keyed by tuple (`m_X,m_Y`) where the rule needs to verify the actor is in the tuple.

**Example:** See `firestore.rules:392-412` (existing tupleProgress rule) and `js/app.js:8480-8482` (existing writeTupleProgress writer). SEC-15-1-03 clones this for `tupleNames.<tk>` and `coWatchPromptDeclined.<tk>`.

### Anti-Patterns to Avoid

- **`memberId == auth.uid` direct comparison** — these have different shapes; will deny every legit write. (See §Q3.)
- **`request.resource.data.<map>.diff(resource.data.<map>)` without property-existence guards** — errors when the map is absent on the existing doc. ALWAYS use `'<key>' in <data> ? <data>.<key> : {}`. (See §Q1.)
- **`.matches(regex)` without anchors** — partial-matches by default; use `^...$`. (See §Q2.)
- **`.toList()` or `.where()` lambda on `rules.Set`** — NOT supported in the Firestore rules expression language as of 2026-04-27. (See Plan 15-01-SUMMARY Auto-fix #2.)
- **Adding actingTupleKey to the writer but NOT the rule's allowlist** — the family-doc 5th branch's `affectedKeys().hasOnly([...])` will reject any write containing a field not in the list. (See §Q7.)
- **Deploying tightened rules BEFORE the writer is updated** — every existing legitimate write breaks. Phase 15.1 deploy order is HOSTING FIRST, RULES SECOND. (See §Q6.)

## Common Pitfalls

### Pitfall 1: `memberId == auth.uid` in the cross-validation rule

**What goes wrong:** Every authed write is denied because `state.me.id` is `'m_<random>'` not `'m_<uid>'`.

**Why it happens:** The 15-REVIEW.md WR-02 §Fix block writes "verify `request.resource.data.memberId == request.auth.uid`" and the 15.1 CONTEXT.md echoes this. Both are wrong about the fixed comparison; the actual link is via the member doc.

**How to avoid:** Use Option C from §Q3 — anchor only, no direct uid comparison. Or Option B (`get()` member-doc proof) if the planner explicitly opts in despite Deferred Ideas.

**Warning signs:** Any new test where a known-good fixture write fails. Run #1-#32 BEFORE shipping the rule change to catch this.

### Pitfall 2: `resource.data.<map>` errors when the map is absent on the existing doc

**What goes wrong:** First-write to a doc that doesn't yet have the `progress`/`tupleNames`/etc. field throws `Property X is undefined` and the rule denies the write.

**Why it happens:** Rules-language null-comparison only handles present-but-null values, not absent fields. (15-01-SUMMARY Auto-fix #3.)

**How to avoid:** Always use `'progress' in resource.data ? resource.data.progress : {}` (or wrap in a `request.resource.data.progress is map` check). The existing tupleProgress + mutedShows rules show the canonical pattern.

**Warning signs:** Tests #28 + #36-style ALLOW tests fail with "Property X is undefined on object" emulator errors.

### Pitfall 3: Forgetting to add `actingTupleKey` to the family-doc 5th branch's allowlist

**What goes wrong:** SEC-15-1-03's writer change adds `actingTupleKey: tupleKeyStr` to the payload, but the existing `hasOnly([...])` allowlist doesn't include it, so the rule rejects.

**Why it happens:** The 5th branch's allowlist is at the OUTER scope; adding a per-key check is at the INNER scope; they're easy to forget about each other.

**How to avoid:** When extending the family-doc 5th branch, add `'actingTupleKey'` to the allowlist in the same edit. Verify via grep: `grep "hasOnly(\[" firestore.rules | grep tupleNames` should show 6 fields not 5.

**Warning signs:** New test #33-style ALLOW for tupleNames+actingTupleKey fails — the OUTER allowlist denies before the INNER per-key check runs.

### Pitfall 4: Source-vs-mirror drift

**What goes wrong:** Tests pass against couch/firestore.rules (source) but production runs ~/queuenight/firestore.rules (mirror) which is out of date — same class as WR-01.

**Why it happens:** `scripts/deploy.sh` does NOT mirror rules. Manual edits to one file aren't mirrored automatically.

**How to avoid:** Every Phase 15.1 commit that touches rules MUST touch BOTH `couch/firestore.rules` AND `~/queuenight/firestore.rules`. A pre-commit grep check or RUNBOOK §H reminder is the safety net.

**Warning signs:** `diff couch/firestore.rules ~/queuenight/firestore.rules` returns non-empty.

### Pitfall 5: Deploying rules before hosting

**What goes wrong:** Rules tightening lands first, in-flight production writers (still on the OLD client code) all break with permission_denied.

**Why it happens:** Phase 15's ritual was rules-first; muscle memory carries over to 15.1; but 15.1 is TIGHTENING (writers must update first), not adding.

**How to avoid:** Encode the reversed deploy ordering explicitly in the close-out plan's `<read_first>` block. See §Q6.

**Warning signs:** Sentry breadcrumbs spike with category `progress.write.failed` or `tupleNames.write.failed` after rules deploy.
