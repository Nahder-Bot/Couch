# Phase 26 — UI Design Contract

> Visual and interaction contract for the position-anchored reactions + async-replay surface. The "Red Wedding" use case: a viewer watching the same title later can experience the original group's reactions aligned to their playback position. Generated by gsd-ui-researcher; verified by gsd-ui-checker.

### 2. Replay scrubber strip

**Visual contract:**
- Single-row strip, `--surface` background, `--space-inline-md` horizontal padding, `--space-stack-sm` vertical padding.
- Container height ≥56px.
- Layout (left to right): 32×32 play/pause icon button (square `--r-sm` radius), 8px gap, `<input type="range">` flex-1, 8px gap, readout text.
- Track: `--surface-3` background (8px tall, `--r-pill` rounded). Filled portion `--brand-grad` from 0 to thumb position. Empty portion `--ink-faint` at 0.4 alpha. (Mirrors Phase 15.5 Wait Up slider treatment per 15.5-UI-SPEC §1.)
- Thumb: 28px circle, `--accent` background, `--shadow-soft`. Active drag scales to 1.1x at `--duration-fast` ease-out.
- Readout: `--t-meta` 13px Inter regular tabular-nums. Format `{m:ss} / {h:mm:ss}` or `{m:ss} / {m:ss}` (auto-collapses hours when total < 1 hr). Current position in `--ink`; total runtime in `--ink-dim`. The `/` separator in `--ink-faint`.

**Range source priority (D-04 default — locked here):**
1. **TMDB title runtime** if available — read from `t.runtime` (movies, minutes) or `t.episode_run_time` (TV, average minutes). Convert to ms.
2. **Fall back to `wp.durationMs`** if Phase 24 broadcast captured it (`wp.durationMs` set when host's player reports a finite `getDuration()`).
3. **Fall back to max observed `runtimePositionMs`** across all replay-able reactions in the wp. Add a 30s cushion so the scrubber doesn't end exactly at the last reaction's position.
4. **Final floor:** if all three fail, use 60 minutes (3,600,000 ms) as a last-resort default. The replay surface should never render with a zero-length scrubber. (Edge case: only `'live-stream'` reactions exist → wp is filtered out by D-10 before this branch fires.)

**Scrubber granularity (Claude's Discretion — locked):**
- **1-second snap.** The `<input type="range">` has `step="1000"` (ms). Default 1-sec snap matches the family use case ("I'm at minute 25, 32 seconds in"). 5-sec snap would feel chunky for the "Red Wedding moment" precision.

**Initial position:** On open, the scrubber starts at `0:00`. The viewer drags to "I'm at minute X" matching their IRL playback. Auto-detection of "where the family was watching at this wall-clock-now" is intentionally out of scope (the viewer is solo on a different timeline).

**Play/pause icon button:**
- 32×32 px. `--surface-2` background, `--r-sm` radius, `--ink` icon color. The icon is a Unicode glyph: `▶` (U+25B6) for paused state, `⏸` (U+23F8) for playing state.
- Tap toggles a local replay clock that advances `runtimePositionMs` at 1x rate (synced to `requestAnimationFrame` for smooth rendering, but the underlying clock uses `Date.now()` deltas to handle tab-blur correctly).
- When playing, the local clock advances and reactions fade in as the position crosses each reaction's `runtimePositionMs` (within drift tolerance — see §3).
- When paused, the local clock holds. Reactions visible at the current position remain visible; new ones do not fade in.
- Pause/resume preserves position. There is no "rewind to 0" affordance — drag the scrubber to 0 manually.

**Drift tolerance (Claude's Discretion — locked):**
- **±2 seconds.** A reaction with `runtimePositionMs = 1500000ms` (25 min) fades in when the local replay clock crosses the window `[1498000ms, 1502000ms]`. 2-second window matches the seed default + matches the family use case (the original reactor had a 1-2s human-typing delay; the viewer's scrubber-drag has a similar tolerance). Sub-second precision is out of scope per CONTEXT § Deferred.
- Implementation: at every animation-frame tick (~60fps), compute `localReplayPositionMs`. Find reactions with `runtimePositionMs` in `[localReplayPositionMs - 2000, localReplayPositionMs + 2000]` AND not yet faded in for this play session. Mount them.

**Drag interaction:**
- `oninput` (continuous during drag): updates the `localReplayPositionMs` and the readout text. Reactions DO NOT fade in during active drag — only on `onchange` (drag end) does the position-aligned set re-evaluate. Rationale: dragging across the scrubber would otherwise rapid-fire mount/unmount every reaction it passes, which is jittery and emotion-flattening.
- `onchange` (drag end): reactions feed re-renders with the position-aligned subset. If the drop position is in a "loud" cluster (many reactions within drift tolerance), all of them mount in quick stagger (60ms cascade — see Motion Catalog).
- Throttle: drag updates throttled to `--duration-fast` (150ms) to avoid 60fps state thrash during rapid drag.

**Aria + screen reader:**
- `<input type="range">` has `aria-label="Move to where you are in the movie"`, `aria-valuemin="0"`, `aria-valuemax="{durationMs}"`, `aria-valuenow="{localReplayPositionMs}"`, `aria-valuetext="{m:ss} of {h:mm:ss}"`.
- Updates on drag-end announce the new position.

### 3. Reactions feed (replay variant)

The replay-variant reactions feed replaces the existing `renderReactionsFeed()` filter logic with a position-aligned subset. Render path lives inside `renderWatchpartyLive` gated on `state.activeWatchpartyMode === 'revisit'`.

**Selection rule (locked):**
- For each reaction `r` in `wp.reactions`:
  - **Skip** if `r.runtimePositionMs == null` (covers `'live-stream'` reactions per D-03 + pre-Phase-26 reactions per D-11 — both have no replay-able position).
  - **Show** if `r.runtimePositionMs ≤ localReplayPositionMs + 2000` (within drift tolerance OR already passed).
  - **Hide** if `r.runtimePositionMs > localReplayPositionMs + 2000` (still in the future relative to viewer's position).
- Sort: ascending by `runtimePositionMs` (chronological in replay time, NOT wall-clock).

**Visual rendering:**
- Same `.wp-reaction-row` recipe as live mode (css/app.css:749-753) — avatar + bubble + emoji + meta. NO visual differentiation by `runtimeSource` (per CONTEXT specifics: "first-class regardless of source").
- Reaction `meta` line under the bubble shows `{member name} · {runtime position formatted}` (e.g., `Aunt Linda · 25:32`). Replaces the live-mode wall-clock format. The position format is `{m:ss}` for parties < 1hr, `{h:mm:ss}` for parties ≥ 1hr — matches scrubber readout.

**Mount animation (per Motion Catalog — see §):**
- New reactions fade in at `--duration-base` (220ms) ease-out, with a subtle 1px `--accent` outer-glow that decays over the same duration. This is the "lit up at the right moment" beat.
- When dragging causes multiple reactions to fall into the visible window simultaneously, stagger mounts at 60ms intervals (capped at 8 — past the 8th, all remaining mount instantly to avoid drag-end stutter).
- Reactions do not unmount when the viewer scrubs backward — once seen, they persist visible. This matches the family-memory framing: revisiting feels additive, not a destructive timeline.

**Empty state (replay variant):**
- When `localReplayPositionMs = 0` and no reactions match (the user just opened the modal, hasn't dragged the scrubber yet): render the empty-message *Nothing yet at this moment.* (italic Instrument Serif, `--ink-dim`, padding 20px, text-align center). Same recipe as the existing live-mode "Reactions will flow in" but with the replay-specific copy.
- When `localReplayPositionMs > 0` but still no reactions match (sparse zone of the title): render the same empty-message. The scrubber strip remains visible above so the user can drag elsewhere.

**Replay-mode reactions COMPOUND (D-05 contract):**
- The footer emoji-row + post-reaction control work IDENTICALLY to live mode. Tapping an emoji in replay variant:
  1. Stamps `runtimePositionMs = currentLocalReplayPositionMs` (where the viewer is on the local replay clock — the play head, NOT the scrubber thumb if they're playing).
  2. Stamps `runtimeSource: 'replay'`.
  3. Writes to `wp.reactions` via `arrayUnion` (existing path).
  4. Optimistic local mount: the new reaction appears in the feed at the viewer's current position with the same accent-glow beat as a position-matched original.
- Future replayers (or even this same viewer scrubbing past their own just-posted reaction in the same session — though uncommon) see this reaction at its `runtimePositionMs`.
