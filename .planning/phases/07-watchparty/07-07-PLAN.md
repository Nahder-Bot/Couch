<objective>
Build the reaction-delay feature that 07-UAT-RESULTS.md Issue #2b diagnosed as net-new (no existing UI, state field, or filter logic). User-stated use case: "don't spoil an upcoming moment — if I'm a few seconds behind the fastest watcher, I want their reactions delayed for me."

**Planner decision, taken autonomously (design-Q from UAT file):** viewer-side delay. Justification:
- Use-case is "spoiler protection for me" → viewer controls their own spoiler window. Each viewer gets to pick their own comfort level independent of others.
- Dramatically simpler implementation: pure render-filter shift in `renderReactionsFeed` at js/app.js:7682-7686. No changes to posting path, no scheduled writes, no cross-viewer timestamp respect.
- Poster-side would require CF-side or client-timeout delayed writes + conflict handling if the poster pauses/leaves during the delay window. Higher risk, later-to-ship, same user outcome for the stated use case.
- If future product asks for poster-side delay as a distinct feature, it can stack on top cleanly (the two aren't mutually exclusive; they'd operate on different dimensions).

**Pattern citation (factual correction from checker):** `setReactionDelay` adopts a pre-await optimistic pattern — mutate local state + render BEFORE the await updateDoc. This matches 07-06's `setWpMode` fix (same rationale: mode-ish toggles need zero-latency feel; rollback on write failure is trivial via onSnapshot). It is DISTINCT from `postReaction`'s post-await echo at `js/app.js:7827-7832` (which awaits the write first, then mutates `wp.reactions`). Do NOT cite postReaction as the precedent — cite 07-06's setWpMode.

Purpose: Ship the advertised control. Currently the footer has NO delay UI whatsoever — the user reported "toggled reaction-delay and saw no change," which is technically correct since the toggle doesn't exist yet. Issue #2 misattributes it to "setting not taking effect" but Issue #2b diagnosed that it's a net-new build.
Output: ~4 surfaces: UI chips in `renderWatchpartyFooter`, `reactionDelay` field in participant shape (create + join write paths), `setReactionDelay` persist fn, filter shift in `renderReactionsFeed`.
</objective>
