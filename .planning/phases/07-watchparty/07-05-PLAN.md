<objective>
Fix Issue #1 from 07-UAT-RESULTS.md: the `watchpartyScheduled` push body renders `startAt` in UTC (GCP Cloud Functions runtime locale), so a party scheduled at 11pm EDT arrives on other family devices labeled "3:00 AM" (~4h offset).

Root cause is confirmed at `queuenight/functions/index.js:169` — `new Date(wp.startAt).toLocaleTimeString([], { hour: 'numeric', minute: '2-digit' })` has no `timeZone` option. Client-side renders via `formatStartTime` (`js/app.js:1763`) are fine (browser uses device locale); the bug is isolated to the CF push body.

Decision (planner, autonomous — Option (c) from UAT file): store creator's IANA timezone at write time on the watchparty doc and pass it as `timeZone` to `toLocaleTimeString` in the CF. Purely additive schema field, no migration risk, most correct semantic (renders in the tz the creator was in when they scheduled — which is what they intended other family members to see).

Purpose: Clean closure of Phase 7 Issue #1. Lifecycle + push delivery already correct — this is pure text-in-body hygiene.
Output: One additive field on the wp doc write path (client), one-line read + pass in CF, redeploy `onWatchpartyCreate`.
</objective>
