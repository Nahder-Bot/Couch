<objective>
Fix two confirmed render-path bugs from 07-UAT-RESULTS.md Issues #2a and #3. Both are structural edits to the watchparty live-modal rendering pipeline in `js/app.js` — data is flowing correctly in both cases; the break is in how (or whether) `renderWatchpartyLive` / `renderReactionsFeed` runs.

**Gap #2a — Hide-reactions toggle has no visible effect.**
- Confirmed at `js/app.js:7864-7873` (`setWpMode`): writes `participants.{uid}.reactionsMode: 'hidden'` to Firestore, then does nothing else. The onSnapshot handler at `js/app.js:3092-3110` only re-renders the banner, not the live modal. Live-modal DOM is stale until the user closes + reopens the modal.
- Fix: adopt a **pre-await optimistic** pattern — local mutation of `state.watchparties[...].participants[uid].reactionsMode` + synchronous `renderWatchpartyLive()` call at the TOP of `setWpMode`, BEFORE the `await updateDoc`. This is distinct from `postReaction`'s **post-await echo** at `js/app.js:7827-7832` (which awaits the write first, then mutates `wp.reactions` and re-renders). We deliberately take the pre-await shape here because (a) the toggle needs zero-latency feel on flaky networks, (b) a failed write will be corrected by the onSnapshot handler within ~1s, and (c) the mutation is a single primitive value — rollback on failure is trivial (snapshot restores).

**Gap #3 — Late joiners see empty reaction feed.**
- Confirmed at `js/app.js:7609`: when `!mine.startedAt`, the branch short-circuits to a "Ready when you are" placeholder and never calls `renderReactionsFeed`. Participants get full reaction data via the existing `onSnapshot` at `js/app.js:3092` (reactions are a doc-array, not a subcollection — the full backlog arrives with the doc), but the render-gate eats it.
- Secondary filter consideration: even if the gate is lifted, the elapsed-mode filter at `js/app.js:7682-7686` (`(r.elapsedMs || 0) <= myElapsed`) would hide everything when `myElapsed == 0` (no startedAt → elapsed is 0). So the fix isn't just "remove the gate" — it's "force wallclock mode for this render when mine.startedAt is absent."
- Fix (Option B from UAT, scoped tight): restructure the `mine && !mine.startedAt` sub-branch **only** to render a compact "Ready when you are" prompt **above** the participant strip + reactions feed (wallclock-forced). The `!mine` branch (viewer not joined) is NOT touched — its existing footer "Join late" CTA is the correct and only action; no new copy added there to avoid a redundant CTA.

Purpose: Close the two major render bugs from Scenario 2 observation. Hide-reactions is a broken user-visible control; late-joiner empty-feed breaks the core social contract of catching up when you arrive mid-party.

<risk_notes>
- **Why pre-await optimistic here, not post-await echo (factual citation correction):** `postReaction` at js/app.js:7827-7832 uses a **post-await echo** pattern — it awaits the Firestore write FIRST, then mutates `wp.reactions` and re-renders. `setWpMode` here deliberately takes the opposite shape: mutate + render BEFORE the await. Rationale: (a) UX — reaction posting can tolerate ~500ms roundtrip for the echo because the user expects a network step to happen for sending a message; mode toggles need to feel instant like a local UI preference. (b) Simplicity of rollback — a single primitive field (`reactionsMode`) is trivially restored by the onSnapshot handler within ~1s if the write fails. (c) Failure mode is visually obvious (the toggle would "snap back" rather than "appear frozen"), which is a better UX than stalling. This is a distinct pattern choice from postReaction's, not an imitation. 07-07's `setReactionDelay` will adopt the same pre-await optimistic shape for the same reasons.
- **Post-"Start my timer" backlog visibility caveat (Gap #3 follow-up):** Once a late joiner taps Start my timer, their reactionsMode defaults to 'elapsed' and `myElapsed` starts at ~0ms. The elapsed filter (js/app.js:7685) will hide reactions where `elapsedMs > myElapsed`. This means the backlog they *were just seeing in wallclock mode* vanishes the instant they press Start, and only catches up as their own timer advances. Depending on product intent this could feel jarring. **Recommended behavior (not required by this plan):** either (a) default late-joiners to 'wallclock' persisted mode when they tap Start my timer on a mid-flight party, or (b) compute effectiveStart = startAt for late joiners so their filter gate opens at the full backlog immediately. Option (b) overlaps directly with Gap #4 (plan 07-08), so defer the decision to that plan. Called out here so it's not a surprise during UAT.
- **Optimistic-mutation race:** `setWpMode`'s optimistic write is overwritten by the onSnapshot listener within ~1s. If the write failed silently (network blip mid-await), the UI would show the user their mode changed when it actually didn't. Acceptable tradeoff for the instant-feedback UX. The alert() in the catch block provides a failure signal in the rare write-rejection case.
