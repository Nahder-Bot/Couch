## Decisions Made

- **Slider `step="0.5"` (not `step="1"`):** Smoother iOS touch-drag feel. The band transform snaps to discrete seconds anyway (5s/15s/60s/300s/900s per band), so the position-side resolution doesn't leak through to Firestore writes. iPhone Safari + Android Chrome both render half-step ranges correctly.
- **WAIT_UP_BANDS at module scope (not inside the transform function):** Searchability via grep + reusability if Plans 03-06 want to query the band table for chip presets that align with band boundaries.
- **`secondsToPosition` returns float (no Math.round):** The only consumer is the slider's `value` attribute, which accepts floats for `type=range`. Rounding here would bias initial thumb placement away from the position the user just set.
- **Defensive `offsetSecClamped` in renderDvrSlider:** Plan 01 added the same clamp inside `setDvrOffset`; this is a second layer at the read boundary. If a corrupt Firestore value sneaks past Plan 01 (e.g., legacy doc with `dvrOffsetMs > 86400000`), the slider initial position still renders within the valid range.
