<decisions>
## Implementation Decisions

### Sport-Mode Slider — Step Ramp

- **D-01:** **Multi-segment step ramp** — slider visually linear from 0 to 86400 sec (24 hr); step granularity adapts by current position so users always have appropriate precision. Bands:
  - 0 - 60 sec: 5-sec steps (preserves current Phase 11 / REFR-10 muscle memory for live-DVR)
  - 60 sec - 5 min: 15-sec steps
  - 5 min - 30 min: 1-min steps
  - 30 min - 2 hr: 5-min steps
  - 2 hr - 24 hr: 15-min steps (or coarser; planner picks final taper)
- Underlying `<input type="range">` cannot vary `step` per-position natively; implementation must transform slider position (0-100) to seconds via a non-linear mapping function. Researcher should investigate if a single transform suffices or if the slider needs to be replaced with a custom touch-aware control.
- The readout text continues to drive from the actual seconds value (existing `dvrReadoutText` pattern at `js/app.js:10306-10310` extends to handle hours).

### Movie-Mode Chip Ladder

- **D-02:** **7 chips + Custom entry**, in this order:

  `Live · 15 sec · 1 min · 5 min · 15 min · 30 min · 1 hr · Custom…`

- These are starting-point shortcuts only. Any value 0 to 24 hr is reachable via Custom picker. Readout always shows actual current value (no chip "snap-to-preset" lying — preserved from current behavior).
- Phone-width consideration: 8 elements (7 chips + Custom) may need wrap-flex CSS; planner verifies fit on iPhone-mini-class viewports.

### Custom Picker — Bottom Sheet Pattern

- **D-03:** **Bottom-sheet modal** — slides up from bottom of viewport when "Custom…" chip tapped or when slider drag would benefit from precision input. Implementation:
  - Reuses existing `.modal` / `.modal-bg` scaffold per `BRAND.md §7` (Screen patterns 3-tier)
  - Body contains: three numeric fields `[hours] [min] [sec]` (mobile keypad-friendly via `inputmode="numeric"` + `pattern="[0-9]*"`) + a row of common-big preset rows (1 hr / 2 hr / 12 hr / 24 hr)
  - Backdrop fades at `--duration-base`; sheet slides at `--duration-deliberate` (BRAND.md §5 motion catalog)
  - `+5 min` nudge button and `Live` reset stay outside the modal — they live next to the chip strip in the live UI so the modal is purely for custom-value entry
- Dismissible by backdrop tap, Cancel button, and successful Submit. Submit copy: "Set Wait Up" or similar friend-voice imperative (planner finalizes copy per BRAND voice rules).

### Past Parties Surface — Tonight Tab Inline Link

- **D-04:** **Tonight tab inline link** — when stale watchparties exist (`5h ≤ (now - startAt) < 25h`), render a single `Past parties (N) ›` link below the active wp banner section. Tap navigates to a dedicated list view.
- When no stale wps exist, the link does not render (zero clutter). The 5h cutoff filter applies to the existing Tonight-tab wp list rendering — researchers should map the current banner / wp-list render path and identify the cleanest insertion point.
- The Past parties list view itself is a new surface but should reuse existing wp-card patterns (poster, title, age, participant count, tap-to-open). Empty state is handled implicitly via the link not rendering.

### Push Template Content Stripping (Cross-Repo)

- **D-05:** Push notification body in `queuenight/functions/index.js` reads each receiver's `participants[receiverId].reactionDelay` at fan-out time. If `> 0`, body uses sender-attributed strip format: `"[SenderName] is reacting to [Title]"`. If `0`, body uses current full-content format. Sender-side behavior unchanged.
- The CF already iterates members for fan-out (Phase 6 push pattern); branching on receiver state per-send is additive, not architectural.
- This change deploys via `firebase deploy --only functions` from `~/queuenight/` per RUNBOOK §H. Cross-repo deploy ordering: queuenight functions BEFORE couch hosting if both ship together (Phase 15.1 reverse-of-Phase-15 ritual still applies).

### Per-Receiver-Only Contract — Reinforcement

- **D-06:** Wait Up offset MUST NOT affect what other viewers see. This is preserved by the existing render filter at `js/app.js:10917` (reads only `mine.reactionDelay`). Phase 15.5 reinforces this in copy: any explanatory tooltip / onboarding text states "Wait Up only changes when YOU see reactions, never anyone else's view." Settings-level explainer is OUT of scope (no Settings UI for Wait Up exists; that's future polish).

### `+5 min` Nudge Placement

- **D-07:** Nudge button renders inline with the active state UI, next to the chip strip (movie mode) or below the slider readout (sport mode). Visible only when `reactionDelay > 0`. Tap adds 300 sec (clamps at 86400 sec / 24 hr max). Same friend-voice copy as the rest of Wait Up surfaces.

### Claude's Discretion

- Exact step-ramp band boundaries beyond the high-level bands above (planner may refine if testing reveals UX issues at boundaries)
- Custom picker submit-button copy (must follow BRAND voice rules — friend-voice imperative, no em-dashes, no tech-mechanism words)
- Past parties list view layout details (poster size, metadata density, sort order — researcher proposes during plan-phase)
- Whether Past parties surface needs its own back-navigation or inherits browser back / native gesture
- Edge case: what happens if a user actively in a wp has it cross the 5h boundary mid-session (most likely answer: keep them in the live banner until they close it; planner verifies)

</decisions>

<specifics>
## Specific Ideas

- **The 7 chips matter as a set, not as fixed values.** User said "don't lock in on my exact values" — chips are starting-point shortcuts only. The contract is: any value 0 to 24 hr is reachable in 1-2 taps (chip tap, or Custom… → numeric entry, or + 5 min nudge from current).
- **`+5 min` nudge replaces a button-of-5-mins-only** — not a configurable amount. Future polish could let users pick the nudge amount; out of scope here. The 5-min default works for the bathroom-bump case (the user's hero example).
- **Live (0-state) is always reachable in one tap** — first chip in the strip. Even when modal is open, dismissing without commit returns to current state; explicit Live tap writes 0.
- **Stale-wp boundary is exact at 5h** since `wp.startAt`. Watchparties at exactly 5h move to Past parties; watchparties at 4h 59m stay on the Tonight banner. Planner verifies edge case for sessions crossing the 5h boundary mid-active-use.
- **Push body when stripped is sender-attributed** (`"Bob is reacting to [Title]"`) not generic (`"Your couch is reacting to [Title]"`) — preserves the social signal that makes the push valuable.

</specifics>
