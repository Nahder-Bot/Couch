# Phase 15.5: Wait Up — Specification

**Created:** 2026-04-28
**Ambiguity score:** 0.1255 (gate: ≤ 0.20)
**Requirements:** 11 locked

## Goal

Convert the existing fixed-cap reaction-delay mechanism into a flexible "Wait Up" feature that supports any per-receiver offset from 0 to 24 hours across both sport-mode and movie-mode watchparties, with a chip + nudge + custom-picker UX. Strip reaction content from server-side push templates when the receiver has Wait Up active so spoiler text stops landing on lock screens before the moment arrives. Add stale-watchparty list hygiene so long-tail catchup parties don't clutter the Tonight surface.

## Background

The Wait Up rebrand (commits `25df50c`, `bb08fd1`, `e0f06c4`, `82fed7c` on 2026-04-27/28) installed the feature voice but not the implementation that voice promises. The underlying render filter at `js/app.js:10917` (`const delayMs = (mine.reactionDelay || 0) * 1000`) supports any value, but the input UIs are capped:

- **Sport-mode slider** (`js/app.js:10290-10310`): `max="180"` seconds, step=5
- **Movie-mode chips** (`js/app.js:11023-11045`): presets `[0, 5, 15, 30]` seconds

The brief's hero examples — *"started Game of Thrones 15 minutes late"* and *"started something 2 hours later"* — fall outside both ranges. Today, Wait Up can hold a friend's reaction for at most 3 minutes.

Push notifications fire server-side at message creation time (`queuenight/functions/index.js`) with full reaction content in the body — meaning a receiver on a 60-second Wait Up offset still gets *"Bob: WHAT A CATCH"* on their lock screen 60 seconds before they see the catch. The render filter holds the in-app render; the push notification spoils anyway.

Watchparties have a 25-hour archive window, but with Wait Up extending offsets to 24 hours, multiple parties can stack as long-tail catchup viewers join over the full day. The Tonight tab's wp list/banner surface needs a freshness cutoff so the active conversation stays visible.

## Requirements

1. **Range expansion — sport mode**: Sport-mode slider supports the full 0 sec to 24 hour Wait Up range.
   - Current: `<input type="range" min="0" max="180" step="5">` at `js/app.js:10297` — caps at 3 minutes
   - Target: Slider scales smoothly across 0 sec to 24 hr; step granularity adapts per range (fine at low values, coarser at high)
   - Acceptance: Drag slider end-to-end and confirm offset value reaches 86400 seconds (24 hr); intermediate stops at 60s, 5m, 15m, 30m, 1hr, 6hr, 12hr land within ±10% of intended values

2. **Range expansion — movie mode**: Movie-mode chip strip + custom picker covers the same 0-24h range.
   - Current: 4 preset chips `[0, 5, 15, 30]` seconds at `js/app.js:11027` — caps at 30 seconds
   - Target: Chip strip with starting-point presets (Live / 15s / 1m / 5m / 15m / 30m / Custom…) plus custom picker for any value up to 24 hr
   - Acceptance: Movie watchparty footer renders ≥6 chips plus a Custom-picker entry; tapping Custom opens a modal that accepts any value 0 sec to 24 hr; confirmed value writes to `participants[me].reactionDelay`

3. **Custom picker form factor**: Custom picker uses three numeric fields (hours / min / sec) plus a common-big-preset row.
   - Current: No custom picker exists — chips are the only input
   - Target: Modal with `[hours] [min] [sec]` numeric inputs (mobile keypad-friendly) plus a row of common-big presets (1 hr, 2 hr, 12 hr, 24 hr)
   - Acceptance: Tapping each numeric field opens the mobile number keypad on iOS Safari + Android Chrome; preset row tap fills all three fields and confirms; modal commits writes `participants[me].reactionDelay` and dismisses

4. **Readout reflects actual value**: The readout label shows the current offset, not the closest preset.
   - Current: Readout text is computed from `dvrReadoutText(sec)` at `js/app.js:10306-10310` — no "snap-to-preset" logic; correct on this dimension already
   - Target: Same — preserve current behavior; ensure picker-set values render correctly via existing `dvrReadoutText` (extend formatting to handle hours)
   - Acceptance: Set offset to 12 min via custom picker; readout reads `12 min` (not `15 min` or other preset); set offset to 1 hr 30 min; readout reads `1 hr 30 min` (or equivalent friend-voice format)

5. **+5 min nudge**: A `+5 min` button is always reachable when Wait Up is active.
   - Current: No nudge affordance exists
   - Target: When `reactionDelay > 0`, render a `+5 min` button alongside the chip/slider control. One tap adds 300 sec to current value; remains reachable until offset reaches the 24h cap
   - Acceptance: With offset at 60 sec, tap `+5 min` button → offset becomes 360 sec; readout updates accordingly; with offset at 23h 56m, tap `+5 min` → offset clamps to 24h max (no overflow)

6. **Live reset**: Tapping `Live` chip / button resets offset to 0 from any non-zero value.
   - Current: Chip/slider value of 0 is the live state but no labeled "Live" affordance exists
   - Target: Always-visible `Live` chip in the chip strip; tap from any state writes 0 to `participants[me].reactionDelay`
   - Acceptance: With offset at 12h, tap `Live` → offset becomes 0; readout reads `Live` or `No wait`; render filter immediately stops holding new reactions

7. **Push template content stripping**: Push notifications strip reaction content when receiver has `reactionDelay > 0`.
   - Current: `queuenight/functions/index.js` push templates include full reaction body in notification text (e.g., `"Bob: WHAT A CATCH"`); fires regardless of receiver offset
   - Target: CF reads receiver's `reactionDelay`; if > 0, sends sender-attributed content-free body (e.g., `"Bob is reacting to [Title]"`); if 0, sends current full-content body
   - Acceptance: Test receiver with `reactionDelay: 30` receives push body `"[SenderName] is reacting to [Title]"` (no reaction content); test receiver with `reactionDelay: 0` receives push body unchanged from current behavior

8. **Per-receiver-only contract**: No surface lets a user's Wait Up offset affect what other viewers see.
   - Current: `participants[mid].reactionDelay` is per-participant; render filter at `js/app.js:10917` reads only `mine.reactionDelay`; correct on this dimension
   - Target: Same — preserve current behavior; reinforce via copy that Wait Up only affects YOUR view, never anyone else's
   - Acceptance: With member A's offset at 1h and member B's offset at 0, member B sees reactions arrive in real time from all senders; member A sees reactions arrive 1h delayed; member B's experience is unaffected by member A's offset (verified via two-device test)

9. **Stale-watchparty list cutoff (5h)**: Watchparties older than 5 hours since `startAt` move off the Tonight tab main wp surface to a secondary "Past parties" surface.
   - Current: Tonight tab wp banner/list surface shows all active watchparties up to the 25h archive window — stack risk grows with Wait Up extending offsets to 24h
   - Target: Tonight tab main surface shows only watchparties where `(now - startAt) < 5h`; older active wps reachable via dedicated "Past parties" entry point (one tap-into access)
   - Acceptance: With 6 active wps at startAt deltas of 1h / 3h / 6h / 12h / 18h / 23h: Tonight tab shows wps at 1h + 3h only (2 wps); the other 4 are reachable via the Past parties surface; tapping into Past parties shows all 4

10. **Secondary "Past parties" surface**: A dedicated surface lists active-but-stale (5h+) watchparties.
    - Current: No such surface exists
    - Target: One-tap entry point on the Tonight tab (label TBD at planning — examples: "Past parties (4)" link, drawer, or chevron) that opens a list of watchparties older than 5h since startAt but still within the 25h archive window
    - Acceptance: Surface renders all wps where `5h < (now - startAt) < 25h` regardless of participant status; each entry shows wp title + age + participant count + tap-to-open action; empty state renders when no stale wps exist

11. **`sw.js` CACHE bump**: PWA cache invalidates on deploy so installed PWAs pick up Wait Up flex changes on next online activation.
    - Current: `sw.js` `CACHE` constant at `sw.js:8` is `couch-v35.1-security-hardening`
    - Target: Bumped to a Phase-15.5-tagged value (likely `couch-v35.5-wait-up` or similar — exact tag chosen at deploy time per RUNBOOK §H)
    - Acceptance: Post-deploy, installed PWAs activate the new sw.js on next online launch; old cache entries deleted; production tag matches the deploy tag passed to `scripts/deploy.sh`

## Boundaries

**In scope:**
- Slider range expansion (sport mode): 180 sec → 24 hr cap, with adaptive step granularity
- Chip strip + custom picker (movie mode): 30 sec → 24 hr cap via Live + 6 preset chips + Custom-picker entry
- Custom picker modal: three numeric fields (hours/min/sec) + common-big preset rows
- `+5 min` nudge button when Wait Up is active
- `Live` reset chip/button for one-tap return to 0
- Push template content stripping in `queuenight/functions/index.js` when receiver `reactionDelay > 0`
- Stale-watchparty list cutoff at 5h since `startAt` (Tonight tab main surface)
- Secondary "Past parties" surface for 5h-25h active wps (entry point + list view)
- `sw.js` CACHE bump
- Brand voice + accessibility correctness (no em-dashes, friend-voice copy, screen-reader-correct aria-labels)

**Out of scope:**
- **Async-replay (persistent reaction archive)** — the "Red Wedding" use case (watch a series years late, feel the original group's reactions). See `seeds/phase-async-replay.md`. Requires schema change (`runtimePositionMs` on reactions), archived-party rejoin flow, runtime-position scrubber. Earns a dedicated phase if Wait Up flex earns its keep in production first.
- **Server-side scheduled push (Option 3 from naming research)** — firing pushes at receiver-specific release time via Cloud Tasks. Adds significant new architecture; deferred until the simpler content-stripping approach has soak time.
- **Title-runtime-position tracker** — manual scrubber for "I'm at minute 23 of the movie." Belongs to the async-replay scope; out of Wait Up flex.
- **Per-user persistent Wait Up default** — the current per-watchparty setting is correct (each session may have different lag). Settings-level "always start with X offset" is a future polish item, not 15.5.
- **Cross-family watchparties** — already out of scope per family-code boundary; not affected by this phase.
- **Banner dismiss / collapse on Tonight tab** — overlapping with Sub-request A in `seeds/phase-async-replay.md`. The Past parties surface absorbs the *list-stacking* problem; the *banner-dismiss-while-active* UX is still seed-material for a future polish phase.

## Constraints

- **Mobile-first PWA.** Custom picker modal must work on iOS Safari standalone PWA + Android Chrome. Numeric inputs must trigger native number keypads (use `inputmode="numeric"` + `pattern="[0-9]*"`).
- **Single-file no-bundler architecture.** No new npm dependencies. Custom picker modal extends existing `.modal` / `.modal-bg` pattern from `BRAND.md` Section 7.
- **Brand voice.** All new copy follows BRAND.md DOes: friend-on-couch register, no em-dashes, sentence-case buttons, no tech-mechanism words ("delay" / "buffer" / "queue" / "offset" / "sync") in user-facing surfaces.
- **Accessibility.** Aria-labels for slider + custom picker + nudge + Live reset; screen-reader-correct value announcement; no banned words in aria-labels.
- **Cross-repo deploy ordering** (RUNBOOK §H): if push template change ships, queuenight `firebase deploy --only functions` precedes couch `firebase deploy --only hosting`. Reverse order risks pre-strip pushes firing against post-strip-aware receivers.
- **Render filter contract preserved.** `participants[mid].reactionDelay` remains the canonical per-receiver render-filter anchor (Phase 7 PARTY-04). No new fields on the reactions object itself; storage stays per-watchparty-per-participant.
- **Per-receiver-only contract.** Wait Up offset MUST NOT affect what other viewers see. This is reinforced both in code (render filter reads only `mine.reactionDelay`) and copy (any explanatory text states the per-receiver scope).
