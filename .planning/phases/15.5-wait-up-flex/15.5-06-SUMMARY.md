## Decisions Made

- **Modal-based Past parties (NOT showScreen destination):** Per RESEARCH ┬з REQ-9, the modal-bg approach avoids the un-highlighted tab-bar state issue (a `showScreen` destination not in the tab list would leave all 5 tabs un-highlighted, breaking the visual hierarchy). The modal also gives free hardware-back / iOS-swipe-back dismissal without modifying app shell routing. UI-SPEC ┬з6b describes the visual as a "Tab-section per BRAND ┬з7 tier 2" тАФ the modal achieves the same visual result (tall full-width sheet with sticky header + list) without routing complications.
- **Cancelled-wp routing in freshWps filter:** Cancelled wps (`wp.status === 'cancelled'`) stay in the banner regardless of age тАФ they only show for 10 minutes per `activeWatchparties()` filter, well under the 5h boundary. Routing them to Past parties would force users to discover a cancellation through a secondary surface, breaking the existing UX. Filter predicate explicitly OR-includes `wp.status === 'cancelled'` in freshWps and AND-excludes in staleWps.
- **`.past-parties-meta` uses `--ink-warm` not `--ink-dim`:** Per UI-SPEC ┬з6b color correction. `--ink-dim` (#847868) on `--surface` (#1c1814) measures 4.0:1 contrast тАФ fails AA Large at 13px text size (Large threshold is 18pt = 24px; 13px is below). `--ink-warm` (#c9bca8) measures 8.5:1 тАФ passes AAA at all sizes. Correction documented in UI-SPEC; applied verbatim in Plan 06.
- **Close button glyph alignment:** Plan 06 plan suggested `├Ч` (U+00D7); plan also specified "if the existing pattern uses a different close glyph, align with convention." Grep'd app.html for existing close glyphs тАФ all 3 existing close buttons use `тЬХ` (U+2715). Aligned with codebase convention.
- **Edge case (mid-session 5h crossover):** Per RESEARCH ┬з REQ-9 + CONTEXT D-04: if a user is actively inside a wp's live modal when it crosses the 5h boundary, the live modal stays open тАФ only the BANNER state recategorizes. On modal close, the wp shows up in Past parties rather than the main banner. No special handling needed because "the live modal doesn't observe banner changes." Verified this behavior by inspection: `closeWatchpartyLive` (js/app.js:10605) doesn't trigger banner re-render; the next `onSnapshot` (4413) or 1-second tick (4426) will route the wp to its correct surface naturally.
