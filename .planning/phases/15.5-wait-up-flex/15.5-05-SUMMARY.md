# Phase 15.5 Plan 05: Cloud Function Reaction-Content Stripping Summary

**Reactions now fan out to every other family member as a push notification, with per-receiver body content stripping based on each receiver's `participants[id].reactionDelay`. When Wait Up is active for a receiver, the push body is `"{senderName} is reacting to {titleName}"` — no spoiler content; when Wait Up is off (live), the body is `"{senderName}: {emoji|text}"` — preserves the social signal pre-15.5 reaction pushes never delivered (since reaction pushes did not exist before this plan).**

## Accomplishments

- **Server-side push gate (REQ-7):** `queuenight/functions/index.js:104-105` — new `reactionPosted: true` key in `NOTIFICATION_DEFAULTS`. Server-side enforcement: `sendToMembers` reads each recipient's `notificationPrefs.reactionPosted` and falls back to this default if unset. Default-ON because reactions are an active engagement signal users expect.
- **Branch 2 — reaction-push fan-out (REQ-7 + REQ-8 completion):** `queuenight/functions/index.js:227-282` — new branch in `onWatchpartyUpdate` that detects `afterReactions.length > beforeReactions.length` and iterates new reactions via `afterReactions.slice(beforeReactions.length)`. For each new reaction: split receivers into `waitUpReceivers` (`reactionDelay > 0`) and `liveReceivers` (`reactionDelay === 0`), then call `sendToMembers` once per group with a different body. Self-echo guard via `excludeMemberId: senderId`. Both calls thread `eventType: 'reactionPosted'` so per-user opt-outs work. Branch returns `null` before falling through to Branch 1's status guard.
- **R-6 burst-append mitigation:** `afterReactions.slice(beforeReactions.length)` iterates ALL new reactions in the diff — not just `[afterReactions.length - 1]`. If a client batches multiple reactions in a single Firestore write window (rare but possible during reconnection or if future code uses `arrayUnion` with multiple elements), every new reaction fires its own pair of `sendToMembers` calls.
- **Defensive coercion on untrusted reaction fields (T-15.5-01):** `senderName`, `titleNameForReaction` are `String(...)`-wrapped + trimmed with safe fallbacks (`'Someone'` / `'your show'`); `kind === 'emoji'` is exact-match only; missing emoji/text body falls back to empty then to a generic `"{senderName} reacted to {titleName}"` to avoid sending `"Bob: "` (orphan colon).
- **Tag-based coalescing:** Both `sendToMembers` calls share the same notification tag `reaction-${wpId}-${reactionId}` — duplicate Firestore re-fires (e.g. CF cold-start retry) don't re-pop the user's lock screen.
- **Client-side DR-3 PLACE 2 (DEFAULT_NOTIFICATION_PREFS):** `js/app.js:124-126` — `reactionPosted: true` mirrors the server default. PWA Settings UI now treats `reactionPosted` as a known event type so opt-out flows work end-to-end.
- **Client-side DR-3 PLACE 3 (NOTIFICATION_EVENT_LABELS, legacy Settings UI):** `js/app.js:160-163` — friend-voice label `'Someone reacts in a watchparty'` with hint `'When someone in your family reacts during a watchparty.'` Both pass BRAND voice + banned-words sweep.
