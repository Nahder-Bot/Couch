## Implementation Approach

### REQ-1 + REQ-4: Sport-Mode Slider Range Expansion + Readout

**The native `<input type="range">` step limitation:** The HTML `step` attribute cannot vary per-position. The locked D-01 approach is a non-linear transform: the slider renders with `min=0 max=100` (a linear 0-100 position), and a `positionToSeconds(pos)` function maps position to seconds using the band boundaries. [VERIFIED: js/app.js:10297 ‚Äî current slider uses `min=0 max=180 step=5` with literal seconds as value]

**Recommended transform function:**

```javascript
// Maps slider position (0-100) to seconds (0-86400)
// Band boundaries at positions: 0, ~7, ~22, ~47, ~67, 100
// (derived from log-like distribution across the 5 bands)
function positionToSeconds(pos) {
  // Band 0-7: 0-60 sec (5-sec steps ‚Üí fine)
  // Band 7-22: 60 sec-5 min (15-sec steps)
  // Band 22-47: 5 min-30 min (1-min steps)
  // Band 47-67: 30 min-2 hr (5-min steps)
  // Band 67-100: 2 hr-24 hr (15-min steps)
  const bands = [
    { pEnd: 7,  sStart: 0,    sEnd: 60,    step: 5   },
    { pEnd: 22, sStart: 60,   sEnd: 300,   step: 15  },
    { pEnd: 47, sStart: 300,  sEnd: 1800,  step: 60  },
    { pEnd: 67, sStart: 1800, sEnd: 7200,  step: 300 },
    { pEnd: 100,sStart: 7200, sEnd: 86400, step: 900 },
  ];
  // ... linear interpolation within band + snap to step
}
function secondsToPosition(sec) { /* inverse */ }
```

The planner must provide the exact band-position boundaries ‚Äî the above are approximate. The key invariant: `positionToSeconds(100) === 86400` and `positionToSeconds(0) === 0`.

**Slider initialization:** On render, `value` attribute must be `secondsToPosition(offsetSec)` (not `offsetSec` directly). The `oninput` handler calls `positionToSeconds(this.value)` to get seconds, then `dvrReadoutText(sec)`.

**Extended `dvrReadoutText`:** Must handle hours. Suggested extension:

```javascript
function dvrReadoutText(sec) {
  if (!sec || sec <= 0) return 'Live';        // UI-SPEC: 'Live' not 'No wait'
  if (sec < 60) return `${sec} sec`;
  if (sec < 3600) {
    const m = Math.floor(sec / 60), s = sec % 60;
    return s ? `${m} min ${s} sec` : `${m} min`;
  }
  const h = Math.floor(sec / 3600), m = Math.floor((sec % 3600) / 60);
  return m ? `${h} hr ${m} min` : `${h} hr`;
}
```

Note: UI-SPEC says the readout displays as italic `"holding {value}"` (sport mode) / `"waiting {value}"` (movie mode). The `dvrReadoutText` function returns the value portion; the surrounding template wraps it in a `<em class="serif-italic">` span.

**`setDvrOffset` clamp:** Add `Math.max(0, Math.min(86400, sec))` at line 11182 equivalent in `setDvrOffset` at 10319-10338.

**Aria-label update:** Change from `"Wait up. How far behind your stream is."` to `"Wait up by, current value {readout}"` (comma not em-dash; dynamic value injected).

### REQ-7: Push Template Content Stripping (Cross-Repo)

**The architectural gap ‚Äî full investigation:**

Reactions are stored as an array in the watchparty doc (`reactions: arrayUnion(reaction)` at `js/app.js:11115`). There is no reaction sub-collection and no existing CF trigger for reactions. The SPEC's description of "push notifications fire server-side at message creation time" describes the TARGET state, not current state.

**Implementation path ‚Äî new branch in `onWatchpartyUpdate`:**

```javascript
// In queuenight/functions/index.js ‚Äî onWatchpartyUpdate, add after the status-to-active branch:

// Branch 2: New reaction appended to reactions array
const beforeLen = (before.reactions || []).length;
const afterLen  = (after.reactions || []).length;
if (afterLen > beforeLen) {
  const newReaction = after.reactions[afterLen - 1]; // last element
  if (!newReaction) return null;
  const senderId = newReaction.memberId;
  const senderName = newReaction.memberName || 'Someone';
  const reactionBody = newReaction.kind === 'emoji'
    ? newReaction.emoji
    : (newReaction.text || '');
  // titleName ‚Äî resolve from after.titleId or after.titleName
  const titleName = after.titleName || 'your show';
  const participants = after.participants || {};
  // Split receivers into two groups by reactionDelay
  const waitUpReceivers    = Object.keys(participants).filter(id => id !== senderId && (participants[id].reactionDelay || 0) > 0);
  const noWaitUpReceivers  = Object.keys(participants).filter(id => id !== senderId && !(participants[id].reactionDelay || 0));
  const tag = `reaction-${wpId}-${newReaction.id || Date.now()}`;
  const url = `/?wp=${wpId}`;
  if (waitUpReceivers.length) {
    await sendToMembers(familyCode, waitUpReceivers, {
      title: senderName,
      body: `${senderName} is reacting to ${titleName}`,
      tag, url
    }, { excludeMemberId: senderId, eventType: 'reactionPosted' });
  }
  if (noWaitUpReceivers.length) {
    await sendToMembers(familyCode, noWaitUpReceivers, {
      title: senderName,
      body: `${senderName}: ${reactionBody}`,
      tag, url
    }, { excludeMemberId: senderId, eventType: 'reactionPosted' });
  }
  return null;
}
```

**New `NOTIFICATION_DEFAULTS` key:** Add `reactionPosted: true` to the object at line 74. Must also be added to the client-side `DEFAULT_NOTIFICATION_PREFS` in `js/app.js:128-155` (the `NOTIFICATION_EVENT_LABELS` / `NOTIF_UI_*` triple ‚Äî DR-3 pattern from Phase 14).

**Race condition risk:** If two participants post reactions simultaneously, both trigger `onWatchpartyUpdate`. Each sees `after.reactions.length > before.reactions.length` and both push. This produces duplicate push notifications for the same reaction. Mitigation: use `newReaction.id` (already present ‚Äî `'r_' + Date.now() + '_' + random`) as the notification `tag` ‚Äî on mobile, same `tag` from two sends coalesces to one notification displayed.

**`onWatchpartyUpdate` double-branch concern:** The existing branch checks `before.status === 'active'` as a guard; the reaction branch fires on every wp update where reactions grew. Both branches can fire in the same invocation (e.g., host starts wp AND another member posts a reaction in the same snapshot ‚Äî unlikely but possible). Use early `return null` correctly: each branch should `return null` at the end of its exclusive processing, OR let them run in sequence with `await` for both. Since they're independent, sequential `await` is safe.

**`after.titleName` vs Firestore lookup:** The `onWatchpartyCreate` trigger resolves title via a Firestore read. For reactions, the wp doc already contains `titleName` (set at creation) ‚Äî use that directly to avoid an extra read per reaction push.

### REQ-9 + REQ-10: Stale-Wp Filter + Past Parties Surface

**5h cutoff math:** `const WP_STALE_MS = 5 * 60 * 60 * 1000`. A wp is "active-fresh" when `(now - wp.startAt) < WP_STALE_MS || wp.startAt > now` (not yet started, or started within 5h). A wp is "active-stale" when `WP_STALE_MS <= (now - wp.startAt) < WP_ARCHIVE_MS`. `activeWatchparties()` is NOT modified ‚Äî both fresh and stale appear in its output.

**Insertion point in `renderWatchpartyBanner()`:** [VERIFIED: `js/app.js:10543-10611`]

```javascript
function renderWatchpartyBanner() {
  const el = document.getElementById('wp-banner-tonight');
  if (!el) return;
  const active = activeWatchparties();
  if (!active.length) { el.innerHTML = ''; return; }
  const now = Date.now();
  const WP_STALE_MS = 5 * 60 * 60 * 1000;
  // Split: fresh wps shown in main banner; stale wps gated behind Past parties link
  const freshWps = active.filter(wp => wp.status !== 'cancelled' && (wp.startAt > now || (now - wp.startAt) < WP_STALE_MS));
  const staleWps = active.filter(wp => wp.status !== 'cancelled' && (now - wp.startAt) >= WP_STALE_MS);
  // Render fresh banners (existing logic)
  const bannerHtml = freshWps.map(wp => { /* existing banner render */ }).join('');
  // Past parties inline link
  const pastPartiesHtml = staleWps.length > 0
    ? `<div class="past-parties-link-row" role="button" tabindex="0" onclick="openPastParties()" onkeydown="if(event.key==='Enter')openPastParties()" aria-label="Past parties, ${staleWps.length} watchpart${staleWps.length===1?'y':'ies'}">
         Past parties <span class="past-parties-count">(${staleWps.length})</span> <span class="past-parties-chevron" aria-hidden="true">‚Ä∫</span>
       </div>`
    : '';
  el.innerHTML = bannerHtml + pastPartiesHtml;
}
```

Note: cancelled wps still show in their 10-min window and should be excluded from both fresh and stale lists (they render separately in the current code ‚Äî verify the filter condition).

**Past parties screen ‚Äî routing approach:**

The app's `showScreen()` function at `js/app.js:4726` matches `document.getElementById('screen-'+name)`. The Past parties surface needs a `<div class="screen" id="screen-past-parties">` in `app.html`. It does NOT get a tab bar entry. Navigation:

## Risk Notes

### PWA / iOS-Specific Risks

**R-1 ‚Äî iOS keyboard + bottom-sheet overlap:** When the custom picker numeric inputs get focus and the iOS software keyboard appears, it can obscure the inputs in a standalone PWA (no browser address bar to push the viewport). Mitigation: After focus, call `input.scrollIntoView({ block: 'center', behavior: 'smooth' })`. The sheet's `max-height: 70vh` should leave room below for the keyboard, but test required on iPhone mini (375px height ~667px physical). [ASSUMED: the standard `scrollIntoView` pattern works in iOS standalone PWA; not verified against live device]

**R-2 ‚Äî iOS `inputmode="numeric"` keypad variation:** On some iOS versions, the numeric keypad includes a decimal point. The `pattern="[0-9]*"` attribute prevents non-numeric submission but doesn't prevent the decimal key from being tapped ‚Äî handle by stripping non-integer characters in the submit handler (`parseInt` already handles this). [ASSUMED based on known iOS Safari behavior]

**R-3 ‚Äî Chip strip wrap on iPhone mini (320px):** At 320px, 8 chips at `min-width: 44px` + `padding: 8px 12px` may produce unusable layout. The UI-SPEC says "wrap is intentional" but validate that the `+5 min` sub-line button is still reachable when the strip wraps to 3 rows.

**R-4 ‚Äî `positionToSeconds` floating-point accumulation:** The step-ramp transform must snap values to the nearest step size within each band (e.g., within the 1-min step band, values must be multiples of 60). Without snapping, intermediate slider positions produce fractional seconds that display awkwardly (`"5 min 7 sec"` instead of `"5 min"`). Add `Math.round(value / step) * step` at each band.

**R-5 ‚Äî `onWatchpartyUpdate` cold-start latency:** Cloud Functions v1 (which this CF uses ‚Äî `functions.firestore.document().onUpdate()`) can have cold-start delays of 1-3 seconds. Reaction pushes will not arrive instantly. This is acceptable for the Wait Up content-stripping use case (the push arriving 2s after the reaction is far better than arriving 60 min after it with spoiler content).

**R-6 ‚Äî Duplicate push on rapid reactions:** If two members post reactions within the same Firestore write window (transactions may batch), the CF may see `afterLen - beforeLen > 1`. The current implementation only handles the last element (`after.reactions[afterLen - 1]`). If multiple reactions were appended, only the last one triggers a push. Mitigation: iterate `after.reactions.slice(beforeLen)` to push for all new reactions. Add rate-limiting via `tag` deduplication.

**R-7 ‚Äî `setReactionDelay` concurrent calls from nudge + chip tap:** If the user taps `+5 min` and a chip quickly, both call `setReactionDelay` which calls `renderWatchpartyLive()` synchronously and then `updateDoc` asynchronously. The second call's optimistic mutation runs before the first Firestore write completes. Since both write to the same field (`participants[me].reactionDelay`), the last writer wins. The throttle in `setDvrOffset` (500ms) prevents this for the slider; `setReactionDelay` does NOT currently have a throttle. For the nudge, add a simple debounce or reuse the `_dvrThrottleHandle` pattern.

### Cross-Repo Deploy Ordering Risks

**R-8 ‚Äî CF before hosting is mandatory for REQ-7:** If couch hosting deploys first (client sees new UI with `reactionDelay > 0` writing larger values), but the CF hasn't deployed yet (no `onWatchpartyUpdate` reaction branch), push notifications fire without content-stripping for any early users who set long Wait Up values. This is a spoiler-delivery window. Follow RUNBOOK ¬ßH strictly: `firebase deploy --only functions` from `~/queuenight` FIRST, then `bash scripts/deploy.sh 35.5-wait-up` from couch repo.

**R-9 ‚Äî NOTIFICATION_DEFAULTS must deploy before client sends `reactionPosted` prefs:** If the client sends `notificationPrefs.reactionPosted = true/false` before the CF knows about `reactionPosted`, the CF's `NOTIFICATION_DEFAULTS` lookup returns `hasDefault: false` ‚Üí defaults to `send`. This is the safe fallback (default-open). The ordering risk is only in the opposite direction (CF sending `reactionPosted` pushes before client Settings UI is ready to show the toggle) ‚Äî but since the Settings toggle for `reactionPosted` is optional polish, the risk is low.
