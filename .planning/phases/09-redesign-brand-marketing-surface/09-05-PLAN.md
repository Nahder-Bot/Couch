`/` serves index.html by default (Firebase Hosting default-routing). No rewrite needed for `/` IF the old queuenight/public/index.html is deleted AND a new landing-serving index.html is placed there — OR a `/` → landing.html rewrite is added explicitly (research option-1, preferred).

Deep-link handling (?invite=, ?claim=):
- Current app expects query params on the root URL.
- After rename: deep links continue to arrive at `/`. landing.html's inline redirect script must forward query params to /app when a deep-link-like param is present — alternative: keep redirect installed-PWA-only, let unauthed deep-linkers see the landing first (probably bad UX for invite recipients).
- Research's guest-invite seed (absorbed in plan 09-07b) says "bootstrapAuth should detect ?invite= BEFORE sign-in gate and route to showInviteRedeemScreen". That implies the ?invite= flow needs to land on /app immediately, not on landing.

Decision (baked into task 1 behavior): landing.html's inline redirect script redirects to /app when ANY of these conditions apply:
1. `window.matchMedia('(display-mode: standalone)').matches` (installed PWA)
2. `window.navigator.standalone === true` (iOS PWA)
3. `window.location.search.includes('invite=')` (invite deep-link)
4. `window.location.search.includes('claim=')` (claim deep-link — Phase 5.8)

All four conditions forward to `/app` + preserve the query string.

Google Fonts + Lighthouse reality: external Google Fonts add a `fonts.googleapis.com` + `fonts.gstatic.com` round-trip that Lighthouse Performance counts against Largest Contentful Paint. With subset-loading + display=swap, landing typically hits 88-92. Hitting ≥95 requires self-hosting Fraunces/Instrument Serif/Inter locally (woff2 in /fonts/) — that's a v2 stretch tracked here but not a Phase 9 requirement. Target for THIS plan: ≥90.

    Step A — Decide routing pattern. Two options per research:

    **Option 1 (recommended, simplest Firebase Hosting behavior):**
    - Keep landing.html as a NAMED file. NOT renamed to index.html.
    - Rename current index.html → app.html.
    - Add Hosting rewrite `"source": "/"` → `"destination": "/landing.html"`. Hosting no longer uses the default index.html lookup.
    - Pros: every file is named after its purpose; no confusing "index is landing".
    - Cons: requires the `/` rewrite (slightly more config than default).

    **Option 2 (alt — simpler hosting, more naming confusion):**
    - Rename current index.html → app.html.
    - Rename landing.html → index.html. Firebase Hosting defaults to serving index.html at `/`, no rewrite needed.
    - Pros: no `/` rewrite; Firebase default works.
    - Cons: "index.html" conceptually means "landing page" now, which contradicts every developer's muscle memory. The file's name no longer tells you what it is.

    Research leans option 1. The `queuenight/firebase.json` edit in task 3 implements the `/` → landing.html rewrite explicitly.

    Execute option 1:
